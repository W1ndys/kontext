# Task
把项目中“所有由 LLM 生成且当前依赖 JSON 输出/JSON 包装/JSON 解析的链路”统一改为“直接输出 YAML 文本并直接存储/解析”，移除 JSON 作为中间传输格式的依赖，同时保证 `.kontext` 物料仍在落盘前经过 YAML 合法性校验，并保持现有 `cmd -> internal/*` 的分层边界不变。

# Task Understanding
本次任务不是单点改模板，而是一次跨 `llm`、`generator`、`packer`、`updater` 的输出协议切换：

- 现状：
  - `internal/llm/openai.go` 的 `ChatStructured` 基于 JSON Schema + `json.Unmarshal`。
  - `internal/generator/parser.go` 中大量 `Parse*` 函数都按 JSON 解析。
  - `internal/generator/engine.go`、`internal/updater/executor.go` 中存在“结构化输出失败 -> 回退到 JSON 模式”的逻辑。
  - 若干模板明确要求模型“严格返回 JSON”。
- 目标：
  - 所有这类链路改成：提示模型直接返回 YAML。
  - 业务层直接从 YAML 文本解析到 Go 结构，或直接把 YAML 文本存盘。
  - 不再要求模型返回 JSON 对象、`{"content":"..."}` 之类 envelope，也不再使用 JSON 代码块/JSON Schema 作为业务协议。
- 影响范围：
  - `generator`：交互式访谈、scan 分析、单文件 YAML 生成、模块契约生成、依赖图等。
  - `packer`：相关文件识别、文件名建议等当前依赖 `ChatStructured` 的结构化子任务。
  - `updater`：更新 YAML 时当前使用 `yamlEnvelope{Content string}` JSON 包裹，需要改为直接返回 YAML 文本。
  - `llm`：需要补一个“YAML 结构化解析”能力，替代或弱化 `ChatStructured` 的 JSON 语义。
- 不应做的事：
  - 不要把 YAML 解析逻辑塞进 `cmd/*`。
  - 不要绕过 `generator.ValidateYAML`、`schema.ValidateBundle`。
  - 不要仅修改模板而保留 JSON fallback 文案与解析代码。

# Project Context
该项目的核心资产本来就是 `.kontext/*.yaml` 与模块契约 YAML，因此让 LLM 直接产出 YAML 更符合工具定位，也能减少：

- JSON 内嵌 YAML 字符串的双层转义；
- `json.Unmarshal` 成功但内部 YAML 实际不可用的中间态；
- updater 中 envelope 解析和二次提取的复杂度。

结合现有架构，这次改动应落在：

- `internal/llm`：统一 LLM 输出协议适配；
- `internal/generator`：统一 YAML 文本提取与 YAML -> struct 解析；
- `internal/packer`：结构化识别结果从 JSON-schema 驱动改为 YAML 直出解析；
- `internal/updater`：取消 JSON envelope，直接拿 YAML 文本做校验和落盘；
- `templates/*`：所有明确要求 JSON 的模板改成 YAML-only 指令。

因为 `schema` 已是 YAML 读取/校验中心，所以生成链路改成 YAML 直出后，后续验证与消费链路仍可复用现有 `yaml.Unmarshal` 体系。

# Relevant Architecture Constraints
- `cmd` 层只做参数解析和流程编排，所有 LLM 输出协议切换必须在 `internal/generator`、`internal/packer`、`internal/updater`、`internal/llm` 内完成。
- `schema` 是 `.kontext` 物料结构定义与校验中心；生成/更新后的 YAML 仍必须遵循 `internal/schema` 中已有结构。
- 所有与模型提供商交互的实现必须收敛到 `internal/llm`，业务流程不能各自散落实现 OpenAI JSON/response-format 细节。
- 所有生成出的 `.kontext` 物料在落盘前都应先进行 YAML 合法性校验；本次切换后这一点更重要，不能因“直接输出 YAML”而跳过。
- `packer` 有上下文预算控制和固定流水线，不应因协议调整改变候选文件扫描、筛选、精筛的既有阶段顺序。
- `update` 仍需保留“计划确认 -> 执行 -> ValidateBundle”闭环，以及备份/回滚准备。
- 禁止静默放行未知 role；若新增 YAML 解析辅助接口，也应继续复用现有 `llm.Message` / `ChatRequest` 契约，不扩散协议复杂度。

# Relevant Module Responsibilities

  - `Client` 接口定义：`Chat` / `ChatStream` / `ChatStructured`
  - `openaiClient.ChatStructured` 的 JSON Schema 约束与 `json.Unmarshal`

  - `.kontext` 物料语义校验；

- 不应承担：
  - OpenAI response-format/JSON Schema 逻辑。

## `internal/packer`
- 当前负责：
  - 文件识别、命名等 LLM 辅助子任务。
- 本次应承担：
  - 改为消费 YAML 结构化输出。
- 不应承担：
  - 手写 provider 级协议细节。

## `internal/updater`
- 当前负责：
  - 根据模板生成更新版 YAML，并校验、备份、写回。
- 本次应承担：
  - 直接接收纯 YAML 内容，不再使用 JSON envelope。
- 不应承担：
  - JSON 包裹兼容长期保留；最多只作为短期兼容 fallback，且应尽快删除。

# Implementation Strategy
## 1. 先在 `internal/llm` 建立 YAML-first 能力
目标是让上层不再依赖 JSON 结构化输出语义。

### 建议改法
在 `internal/llm/types.go` 中调整 `Client` 接口，新增一个明确表达 YAML 语义的方法，例如：

- 保留 `Chat`
- 保留 `ChatStream`
- 用 `ChatYAML(req *ChatRequest, out any) (*ChatResponse, error)` 替代或补充 `ChatStructured`

如果为了减少大面积改名，也可以暂时保留 `ChatStructured` 方法名，但实现语义必须改为“请求 YAML 并按 YAML 解析”；不过从可维护性看，更推荐显式新增 `ChatYAML`，再逐步替换调用点。

### 在 `internal/llm/openai.go`
实现一个 YAML 版本解析流程：

- 直接调用普通 `createChatCompletion(..., openai.ChatCompletionNewParamsResponseFormatUnion{})`
- 不再构造 `OfJSONSchema`
- 读取 `choice.Message.Content`
- 先 strip markdown code block
- 再用 `yaml.Unmarshal([]byte(cleaned), out)`

建议在 `internal/llm/openai.go` 或新增 `internal/llm/yaml.go` 中抽出辅助函数：

- `stripYAMLCodeBlock(content string) string`
- `unmarshalYAMLResponse(content string, out any) error`

并统一错误文案：
- `"解析 YAML 输出失败"`
- `"响应内容为空"`
- 不再出现 `"生成 JSON Schema 失败"`、`"解析结构化输出失败"` 这类 JSON 术语。

### 处理重试辅助
`internal/packer/identifier.go`、`internal/packer/namer.go`、`internal/updater/executor.go` 当前依赖 `llm.ChatStructuredWithRetry`。若该函数已存在于其他文件（虽然上下文未展示），需要同步：
- 改其内部调用为 `ChatYAML`
- 或新增 `ChatYAMLWithRetry`

推荐新增：
- `ChatYAMLWithRetry(client Client, req *ChatRequest, out any, maxRetries int, onRetry ...)`

并逐步把调用点替换过去，避免名字误导。

## 2. 统一 `generator` 的解析器为 YAML 解析
在 `internal/generator/parser.go` 做一次集中替换。

### 具体修改
- 删除 `encoding/json` 引入，改用 `go.yaml.in/yaml/v4`
- `stripCodeBlock` 改为兼容：
  - ```yaml
  - ```yml
  - ```json（为了兼容历史响应，可不限制语言）
  - 无语言标记的 ```
- 所有 `Parse*` 函数改为 `yaml.Unmarshal`

示例方向：
- `ParseInterviewResponse(raw string)`
- `ParseGeneratedYAML(raw string)`
- `ParseAnalyzedFiles(raw string)`
- `ParseSelectedFiles(raw string)`
- `ParseSingleFileYAML(raw string)`
- `ParseModuleContractYAML(raw string)`

### 降级策略
`ParseInterviewResponse` 目前 JSON 失败会退化成纯文本问题，这个策略应该保留：
- YAML 解析失败时，仍返回：
  - `Type: "question"`
  - `Question: raw`
  - 默认选项

这符合交互式 init 的鲁棒性要求。

## 3. 修改 `internal/generator/engine.go` 的所有 JSON 文案与 fallback
这是业务侧最明显的改动点。

### `runInterview`
当前最大轮数后的消息是：
- “请以 JSON 格式回复：...”

要改成类似：
- “请以 YAML 格式直接回复，例如：
  ```yaml
  type: done
  summary: |
    ...
  ```”

### `interviewStep`
- `client.ChatStructured` 改为 `client.ChatYAML` 或等价 YAML 方法
- 错误文案改为：
  - `"YAML 结构化输出失败"`
  - `"回退解析 YAML 失败"`

### `GenerateStructuredYAML`
- 改名更贴切，如 `GenerateProjectYAMLBundle`
- 优先走 `ChatYAML`
- fallback 不再叫 `legacy JSON`
- second try 仍用 `Chat`，但要求“直接返回合法 YAML”

### `generateLegacyYAML`
建议重命名为：
- `generateYAMLWithRetry`
或
- `generateProjectYAMLBundleFallback`

重试提示改为：
- “上次返回的 YAML 格式不正确，错误: ... 请重新生成，直接返回合法 YAML，不要使用 JSON，不要添加额外说明。”

### `AnalyzeProjectFiles` / `SelectKeyFiles` / `GenerateDependencyGraph`
- 先走 `ChatYAML`
- fallback 用 `Chat`
- fallback 解析函数对应改为 YAML 解析
- 错误信息去掉“传统 JSON 解析”

### `ParseModuleDependencyGraph`
- 从 JSON 改成 YAML 解析
- 函数注释同步改为“解析模块依赖关系图的 YAML 响应”

### `GenerateSingleYAML`
这是最关键的“直接输出并直接存储”链路之一，应进一步简化：
- 去掉“尝试解析 JSON 响应”
- 优先：
  - `ChatYAML` 解析到 `SingleFileYAML`
  - 校验 `structured.Content`
- fallback：
  - 直接把 `resp.Content` 及 `extractYAMLFromResponse(resp.Content)` 作为 YAML 候选
- 错误文案里不再出现“JSON 响应中的 YAML 不合法”

### `GenerateModuleContract`
同样把：
- `ChatStructured` -> `ChatYAML`
- `ParseModuleContractYAML` -> YAML 解析
- 重试提示 -> 直接返回 YAML

## 5. 重构 `updater`：去掉 JSON envelope，直接接收 YAML 文本
`internal/updater/executor.go` 需要显著简化。

### 删除内容
- 删除 `encoding/json`
- 删除：
  - `type yamlEnvelope struct { Content string ... }`
  - `parseYAMLEnvelope`
  - `generateYAMLEnvelopeLegacy`

### 重写 `generateYAMLEnvelope`
建议直接改名为：
- `generateYAMLContent`

实现逻辑：
1. 直接 `llm.ChatWithRetry(e.client, req, 3, nil)` 获取文本；
2. 使用 `generator` 中已有的 YAML 提取辅助（若可复用）或本文件内部 strip code block；
3. 返回清理后的纯 YAML 文本；
4. 若为空或无效，在 `generateContent` 中追加 assistant/user 重试消息，要求“直接返回合法 YAML”。

也可保留一条“结构化 YAML 映射”的轻量路径，但 updater 实际只需要最终 YAML 文本，不需要 envelope，因此没必要再映射到 `{content: ...}`。

### `generateContent`
把：
- `resp, content, err := e.generateYAMLEnvelope(...)`
改为：
- `resp, content, err := e.generateYAMLContent(...)`

重试消息改为：
- `"上一次返回的 YAML 无法解析：%v。请保持最小修改，并直接返回完整、合法的 YAML 文本；不要使用 JSON，不要包装在 content 字段中。"`

### `applyAction`
保持不变，因为它本来就是直接写文本：
- `fileutil.WriteFile(targetPath, []byte(content))`

这正好符合“直接存储，无需使用 json 进行传输解析”。

### 模板改写原则
统一要求模型：
- 直接输出 YAML
- 不要 JSON
- 不要额外解释
- 如果是“对象型结果”，给 YAML 示例
- 如果是“单文件正文”，明确“直接输出该 YAML 文件完整内容”

### `templates/init_generate_system.tmpl` 推荐输出形态
与其让模型返回 JSON 对象里包字符串，不如让它直接返回一个总 YAML 对象，例如：
```yaml
project_manifest: |
  project:
    ...
architecture_map: |
  layers:
    ...
conventions: |
  coding:
    ...
module_contracts:
  cmd: |
    module:
      ...
  packer: |
    module:
      ...
```
这样 `GeneratedYAML` 仍可保持字符串字段，业务改动较小，但传输格式已经是 YAML。

如果希望进一步自然化，也可以让模型输出嵌套 YAML 对象而不是 block scalar；但这会要求 `GeneratedYAML` 类型同步变化，改动更大。为了控制范围，优先采用“YAML 外层对象 + 内层文件内容字符串”的方案。

# Notes and Guardrails
- 优先做“YAML-first + 必要最小兼容”，不要做“继续保留 JSON 主路径，YAML 只是附加选项”。否则任务目标无法真正达成。
- 若短期必须保留兼容 fallback，也只能是：
  - 首选 YAML；
