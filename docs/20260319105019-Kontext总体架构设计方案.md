# 文档一：Kontext 总体架构设计方案（修订版 / LLM 增强 Pack MVP）

## 1. 项目概述

### 项目名称

Kontext

### 项目定位

Kontext 是一个面向 AI 协作开发的命令行工具。它的核心目标不是直接替代开发者完成整个项目，而是：

> 将项目知识、架构约束、模块边界、编码规范和当前任务意图，整理并压缩成一份适合大模型直接阅读和执行的高质量 Markdown Prompt 文档。

与前一版不同，本版 MVP 明确引入 **LLM 调用**，用于提升 `pack` 产物质量，而不是只依赖规则拼装。

### 一句话定义

**先用 LLM 帮你把项目和任务“翻译”成高质量开发上下文，再把这份上下文保存成可复用的 Markdown Prompt 文档。**

---

## 2. 为什么在 MVP 阶段引入 LLM

前一版思路中，`pack` 主要依赖静态规则、关键词匹配和模板拼装。这样实现简单，但有几个明显问题：

- 无法准确理解自然语言任务意图
- 难以判断任务真正涉及哪些模块
- 输出往往是“信息堆叠”，而不是“高质量 Prompt”
- 缺少任务级摘要、开发建议和约束强化
- 不能把散乱项目资料重组成适合模型执行的上下文

因此本版设计调整为：

> **MVP 阶段就引入 LLM，但把 LLM 放在“理解任务、筛选上下文、生成最终 Prompt”这三个关键环节上。**

这样做的价值是：

1. 明显提高最终 Prompt 质量
2. 更适合真实使用场景
3. 更早验证 Kontext 的核心价值，而不是只验证一个静态拼接器
4. 更符合你“让模型帮自己开发自己”的自举目标

---

## 3. 当前阶段产品目标

当前阶段仍然聚焦一个核心能力：`pack`

但本版 `pack` 的目标升级为：

> 输入一句开发任务描述，结合 `.kontext/` 物料、部分代码结构和 LLM 分析，生成一份高质量的 Markdown Prompt 文档，并保存到文件中，供后续大模型直接阅读使用。

### 输入

- 一句任务描述
- `.kontext/` 基础物料
- 当前项目部分代码与目录结构
- LLM 配置（API Key / Base URL / Model）

### 输出

- 一份保存在磁盘上的 Markdown Prompt 文件，例如：

```bash
.kontext/prompts/2026-03-19_implement-validate-command.md
```

### 输出文档内容应包含

- 当前任务说明
- 项目摘要
- 涉及模块分析
- 架构边界和开发约束
- 相关模块职责
- 推荐修改文件
- 必要源码片段
- 明确的实现要求
- 给大模型的执行方式说明

---

## 4. 当前阶段范围边界

### 本阶段包含

- `kontext init`
- `kontext pack`
- `kontext validate`
- 最小 `.kontext/` 物料体系
- LLM 驱动的任务分析与 Prompt 生成
- 将最终 Prompt 保存为 Markdown 文件
- 终端打印生成文件路径

### 本阶段不包含

- 自动更新 `.kontext/`
- 从代码全量逆向生成所有物料
- 多轮 agent orchestration
- 剪贴板复制
- IDE 插件
- 团队协作服务端
- 自动写代码并执行

---

## 5. 核心设计思路

本版 MVP 的核心不是“把物料拼起来”，而是：

> **把静态项目知识 + 当前任务意图 + 代码结构线索，交给 LLM 整理成一份更强、更稳、更适合继续编码的大模型输入文档。**

因此 `pack` 的真正职责是：

1. 加载项目上下文
2. 初步筛选候选信息
3. 调用 LLM 做任务级理解与重构
4. 生成高质量 Prompt 文档
5. 保存为 Markdown 文件

---

## 6. 总体架构设计

### 顶层目录结构

```text
cmd/ CLI 命令层
internal/schema/ .kontext 物料结构定义与加载
internal/packer/ pack 主流程
internal/llm/ LLM 调用封装
internal/fileutil/ 文件读写与扫描
internal/promptdoc/ Markdown Prompt 文档保存
templates/ LLM 提示模板 / Prompt 模板
```

### 模块职责概览

#### `cmd/`

负责：

- 参数解析
- 命令编排
- 输出结果路径和错误信息

不负责：

- 任务分析
- 文档生成
- 物料加载细节

#### `internal/schema/`

负责：

- `.kontext/` YAML 结构定义
- 统一加载 `.kontext/` 内容
- 提供 Bundle 数据结构

#### `internal/fileutil/`

负责：

- 读写文件
- 读取代码片段
- 获取目录结构摘要
- 判断文件存在性

#### `internal/llm/`

负责：

- 封装对 OpenAI-compatible API 的调用
- 接收 system/user prompt
- 返回分析结果与生成内容

#### `internal/packer/`

负责：

- pack 核心流程编排
- 基础规则匹配
- 候选上下文收集
- 构造 LLM 输入
- 接收 LLM 输出
- 生成最终 Prompt 文档内容

#### `internal/promptdoc/`

负责：

- 生成 Markdown 文件名
- 保存最终 Prompt 到 `.kontext/prompts/`
- 管理输出目录

---

## 7. .kontext 物料设计

当前阶段仍然使用最小 4 类物料。

```text
.kontext/
├── PROJECT_MANIFEST.yaml
├── ARCHITECTURE_MAP.yaml
├── CONVENTIONS.yaml
├── module_contracts/
│ ├── cmd_CONTRACT.yaml
│ ├── schema_CONTRACT.yaml
│ ├── packer_CONTRACT.yaml
│ └── fileutil_CONTRACT.yaml
└── prompts/
```

### 各物料职责

#### `PROJECT_MANIFEST.yaml`

描述项目是什么：

- 项目名称
- 项目目标
- 技术栈
- 核心命令
- 非目标范围

#### `ARCHITECTURE_MAP.yaml`

描述项目结构：

- 目录职责
- 分层边界
- 架构规则

#### `CONVENTIONS.yaml`

描述开发规范：

- 编码约束
- 错误处理约束
- 禁止行为
- AI 修改要求

#### `module_contracts/*.yaml`

描述模块边界：

- 模块职责
- owns / not_responsible_for
- depends_on
- 允许改动方式

#### `.kontext/prompts/`

保存由 `pack` 生成的 Markdown Prompt 文档

## 8. Pack 核心工作流（修订版）

### 总流程

```text
输入任务描述
→ 加载 .kontext 物料
→ 扫描必要项目结构和候选源码
→ 初步匹配相关模块
→ 构造 LLM 分析输入
→ LLM 分析任务并生成高质量 Prompt 文本
→ 保存 Markdown 文件
→ 返回文件路径
```

### 详细阶段

#### 1. Load Bundle

从 `.kontext/` 中读取：

- Manifest
- Architecture
- Conventions
- Contracts

形成统一数据结构 `Bundle`

#### 2. Discover Candidate Context

通过轻量规则收集候选上下文：

- 匹配相关 contract
- 匹配候选源码文件
- 读取目录结构摘要
- 收集通用架构与规范内容

注意：这一步不是最终拼接，只是为 LLM 提供素材。

#### 3. Build LLM Input

将以下内容送入 LLM：

- 当前任务
- 项目摘要
- 架构规则
- 编码规范
- 模块契约
- 候选代码片段
- 希望输出的 Markdown 文档结构

#### 4. LLM Generate Prompt Document

由 LLM 输出一份高质量 Markdown 文档，要求：

- 结构稳定
- 语言清晰
- 约束明确
- 便于后续大模型继续开发
- 避免空话和泛泛建议

#### 5. Save Markdown

将生成结果保存到：

```text
.kontext/prompts/<timestamp>_<task-slug>.md
```

#### 6. Return Result

CLI 输出：

- 文件保存路径
- 可选显示摘要
- 若失败则返回错误原因

---

## 9. 为什么保存为 Markdown 文件，而不是剪贴板

本版明确取消剪贴板方案，改为输出 Markdown 文件，原因如下：

1. **可复用**

- 生成一次，可以重复给不同模型使用

2. **可版本化**

- 可纳入 git 或留作历史记录

3. **可审阅**

- 方便人工检查 Prompt 质量

4. **可沉淀**

- Prompt 本身会成为项目的一部分知识资产

5. **更适合自举**

- Kontext 开发自己时，可以持续积累 prompt 文档

因此，MVP 的核心产物不再是“一段临时复制文本”，而是：

> **一个可持续复用的 Prompt 文档资产。**

---

## 10. LLM 在系统中的职责

### LLM 负责

- 理解任务意图
- 判断任务相关上下文
- 将上下文压缩为更高质量的开发 Prompt
- 输出结构化 Markdown 文档

### LLM 不负责

- 直接执行代码修改
- 自动读完整个仓库
- 自动维护 `.kontext/`
- 替代 schema 的结构化事实存储

### 设计原则

Kontext 不是“完全靠 LLM 自由发挥”，而是：

> 用 `.kontext/` 提供可靠事实，用 LLM 做高质量理解和重组。

这能兼顾：

- 稳定性
- 灵活性
- 生成质量

---

## 11. LLM 调用设计

### 配置方式

通过环境变量或配置文件提供：

- `KONTEXT_LLM_BASE_URL`
- `KONTEXT_LLM_API_KEY`
- `KONTEXT_LLM_MODEL`

### MVP 支持

先只支持一个 OpenAI-compatible provider 即可。

### 调用方式

`internal/llm` 提供统一接口，例如：

```go
type Client interface {
Generate(ctx context.Context, req GenerateRequest) (GenerateResponse, error)
}
```

### GenerateRequest 内容

- system prompt
- user prompt
- model
- temperature
- max tokens

---

## 12. Prompt 文档输出规范

LLM 最终输出的 Markdown 文档建议固定结构：

```markdown
# Task

...

# Task Understanding

...

# Project Context

...

# Relevant Architecture Constraints

...

# Relevant Module Responsibilities

...

# Relevant Files

...

# Implementation Strategy

...

# Notes and Guardrails

...
```

### 目标

让这份 Markdown 文件可直接作为“下一位大模型”的输入材料。

---

## 13. 技术选型

- 语言：Go
- CLI：cobra
- YAML：go.yaml.in/yaml/v4
- 模板：text/template
- HTTP：net/http
- 日志：可先不加或后续加 slog
- 测试：testing

### 额外说明

MVP 阶段不追求多 provider，不做复杂 router，先跑通一条最强主链路。

---

## 14. 非功能要求

1. 默认本地运行
2. LLM 调用失败时要给出明确错误
3. 输出文件命名稳定、可追踪
4. 生成结果可审阅、可复用
5. 不把复杂业务逻辑塞进 CLI 层
6. 保持为后续 `generate` / `update` 留扩展空间

---

## 15. 阶段验收标准

### 功能验收

- `kontext init` 能生成 `.kontext/` 基础模板
- `kontext validate` 能校验 YAML 合法性
- `kontext pack "<task>"` 能生成 Markdown Prompt 文件
- CLI 能输出生成文件路径
- LLM 调用链路可用

### 产品验收

- 生成的 Prompt 文档质量明显高于静态拼接
- 文档适合直接交给大模型继续编码
- 能用于开发 Kontext 自己的下一步功能
- 至少完成 3 次真实自举任务

---

## 16. 后续演进方向

本版 MVP 完成后，可以演进：

- `generate`：从需求或代码生成 `.kontext/`
- `update`：代码变更后同步更新物料
- `pack --from-file`：从任务说明文档生成 Prompt
- `pack --json`：同时输出结构化元数据
- 更智能的源码筛选和上下文裁剪
- 多 provider 支持
