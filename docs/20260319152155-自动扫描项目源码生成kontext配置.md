# 自动扫描项目源码生成 .kontext 配置

## Context

当前 `kontext init "描述"` 的 AI 模式需要用户手动输入项目描述并多轮对话，无法自动感知项目实际代码。需要新增一个模式：自动扫描当前项目源码（目录结构、依赖文件、关键代码），将扫描结果喂给 LLM，一次性生成 `.kontext/` 下的三个 YAML 文件。

## 命令接口

```bash
kontext init --scan        # 自动扫描项目源码并生成
```

在 `init` 命令上添加 `--scan` flag，复用已有的覆盖确认逻辑。

## 扫描内容

利用已有的 `fileutil.ScanDirectoryTree()` + `fileutil.ReadCodeSnippets()` 收集：

1. **目录树**（深度 5）
2. **依赖/配置文件全文**：go.mod, package.json, Cargo.toml, pyproject.toml, pom.xml, build.gradle, Makefile, Dockerfile, .gitignore 等
3. **入口及关键源码前 50 行**：筛选入口文件（main.go, main.py, index.ts 等）和核心源码文件，限制总文件数最多 30 个

## 文件变更清单（2 新建 3 修改）

### 新建

| 文件 | 作用 |
|------|------|
| `templates/init_scan_system.tmpl` | 扫描模式的系统提示词（含 YAML schema，指导 LLM 从源码分析生成配置） |
| `templates/init_scan_user.tmpl` | 用户消息模板（渲染目录树 + 依赖文件内容 + 代码片段） |

### 修改

| 文件 | 变更 |
|------|------|
| `templates/templates.go` | 新增 `InitScanSystem` / `InitScanUser` 两个 embed 变量 |
| `internal/generator/engine.go` | 导出 `ValidateYAML`、`RenderTemplate`；提取 `WriteGeneratedYAML(generated *GeneratedYAML)` 供 `runScanInit` 复用 |
| `cmd/init.go` | 添加 `--scan` flag；新增 `runScanInit()` 函数：扫描项目 → 渲染 prompt → 单次 LLM 调用 → 解析 + 校验 + 写入 |

## 关键复用

- `fileutil.ScanDirectoryTree()` — 目录扫描（`internal/fileutil/scanner.go`）
- `fileutil.ReadCodeSnippets()` — 代码片段读取（`internal/fileutil/scanner.go`）
- `generator.ParseGeneratedYAML()` — JSON 解析 LLM 响应（`internal/generator/parser.go`）
- `generator.ValidateYAML()` — YAML 合法性校验（导出后）（`internal/generator/engine.go`）
- `generator.WriteGeneratedYAML()` — 目录创建 + 文件写入（提取后）（`internal/generator/engine.go`）
- `init_generate_system.tmpl` 中的 YAML schema 定义 — 在 `init_scan_system.tmpl` 中复用

## 实现顺序

1. `templates/init_scan_system.tmpl` + `init_scan_user.tmpl`
2. `templates/templates.go` — 添加 embed
3. `internal/generator/engine.go` — 导出函数 + 提取 WriteGeneratedYAML
4. `cmd/init.go` — 添加 `--scan` flag + `runScanInit()`
5. `go build && go vet` 验证

## 验证

1. `go build && go vet ./...`
2. 在当前项目下运行 `./kontext init --scan`，确认生成的 3 个 YAML 文件内容合理
3. 确认原有 `kontext init` 和 `kontext init "描述"` 不受影响
