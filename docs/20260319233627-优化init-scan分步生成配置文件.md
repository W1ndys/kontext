# 优化 init --scan：分步生成 + 断点续传

## Context

当前 `init --scan` 存在以下问题：

1. **超时风险高**：一次性生成 4 类物料（PROJECT_MANIFEST + ARCHITECTURE_MAP + CONVENTIONS + module_contracts），LLM 处理时间过长
2. **用户体验差**：长时间等待（240秒+）只有一个转圈动画，没有进度反馈
3. **失败代价高**：任何一个文件生成失败，全部需要重试
4. **结构化输出不稳定**：`map[string]string` 类型的 `module_contracts` 在某些 LLM 的结构化输出中可能有兼容性问题
5. **API 中断全部白费**：如果在阶段6 LLM 调用时 API 断开（EOF、网络超时等），前面阶段1-5的扫描、分析、选择等工作全部丢失，必须从头开始

## 优化方案比较

| 方案                      | 优点                     | 缺点                   |
| ------------------------- | ------------------------ | ---------------------- |
| **A. 串行分步生成**       | 进度清晰、失败可部分恢复 | 总耗时长（无法并行）   |
| **B. 全部并行生成**       | 速度最快                 | 文件之间缺乏一致性参考 |
| **C. 混合流水线（推荐）** | 速度与质量兼顾           | 实现稍复杂             |

### 推荐方案：混合流水线

```
Step 1: 生成 PROJECT_MANIFEST.yaml（基础信息，其他文件参考它）
        ↓
Step 2: 并行生成 ARCHITECTURE_MAP.yaml + CONVENTIONS.yaml
        （这两个相对独立，可参考 Step 1 的结果）
        ↓
Step 3: 生成 module_contracts/
        （需要参考架构分层信息，每个模块一个文件，可并行）
```

**核心思路**：

- Step 1 先出基础，确保项目名称、技术栈等关键信息统一
- Step 2 架构和规范可以并行，互不依赖
- Step 3 模块契约需要架构信息作为上下文，但各模块之间可并行

### 4. 修改 `cmd/init.go` 中的阶段6

将当前的单步改为三个子步骤：

```
🤖 阶段 6/8：生成项目清单...
   ✓ PROJECT_MANIFEST.yaml (8.2 秒)

🏗️ 阶段 7/8：生成架构与规范... (并行)
   ✓ ARCHITECTURE_MAP.yaml (6.1 秒)
   ✓ CONVENTIONS.yaml (5.8 秒)

📦 阶段 8/8：生成模块契约... (5 个模块并行)
   ✓ cmd_CONTRACT.yaml (4.2 秒)
   ✓ config_CONTRACT.yaml (3.9 秒)
   ✓ packer_CONTRACT.yaml (4.5 秒)
   ✓ schema_CONTRACT.yaml (4.1 秒)
   ✓ llm_CONTRACT.yaml (3.8 秒)
```

### 5. 并行执行策略

Step 2 和 Step 3 使用 goroutine 并行：

```go
// Step 2 并行
var wg sync.WaitGroup
var archResult, convResult string
var archErr, convErr error

wg.Add(2)
go func() {
    defer wg.Done()
    archResult, archErr = GenerateSingleYAML(client, archSystemPrompt, archUserMsg)
}()
go func() {
    defer wg.Done()
    convResult, convErr = GenerateSingleYAML(client, convSystemPrompt, convUserMsg)
}()
wg.Wait()
```

Step 3 对每个模块也并行，但限制最大并发数（如 3），避免 API rate limit：

```go
sem := make(chan struct{}, 3) // 最大并发数 3
for _, mod := range modules {
    wg.Add(1)
    go func(m string) {
        defer wg.Done()
        sem <- struct{}{}
        defer func() { <-sem }()
        // 生成单个模块契约
    }(mod)
}
wg.Wait()
```

## 优化二：断点续传 / 检查点恢复机制

### 问题分析

当前流程中，如果在阶段 6-8（LLM 生成配置文件）时 API 中断（EOF、网络超时、rate limit 等），用户会看到：

```
   ⠙ 识别技术栈... (384秒)
Error: 结构化输出失败: 调用结构化输出失败: Post "...": unexpected EOF
```

此时阶段 1-5 的所有中间结果（目录扫描、AI 文件识别、配置读取、概要提取、重点文件选择）全部丢失，下次运行必须从头开始。

### 解决方案：检查点缓存

在 `.kontext/.cache/` 目录下保存各阶段的中间结果，支持断点续传。

#### 缓存结构

```
.kontext/
├── .cache/
│   ├── scan_checkpoint.json    # 检查点元数据
│   ├── stage1_files.json       # 阶段1：扫描到的文件列表
│   ├── stage2_analyzed.json    # 阶段2：AI 识别的配置/源码文件
│   ├── stage3_configs.json     # 阶段3：读取的配置文件内容
│   ├── stage4_summaries.json   # 阶段4：提取的文件概要
│   ├── stage5_selected.json    # 阶段5：选择的重点文件
│   └── stage6_keyfiles.json    # 阶段6准备：重点文件内容
```

#### 检查点元数据（scan_checkpoint.json）

```json
{
  "version": "1.0",
  "created_at": "2026-03-19T23:36:27+08:00",
  "project_hash": "a1b2c3d4", // 项目目录结构的 hash，用于判断缓存是否过期
  "completed_stages": [1, 2, 3, 4, 5],
  "current_stage": 6,
  "generated_files": {
    "PROJECT_MANIFEST.yaml": true,
    "ARCHITECTURE_MAP.yaml": false,
    "CONVENTIONS.yaml": false
  }
}
```

### 新增命令行参数

```bash
# 正常运行（自动检测缓存，如果存在有效缓存则询问是否继续）
kontext init --scan

# 强制从头开始（忽略缓存）
kontext init --scan --fresh

# 强制使用缓存继续（不询问）
kontext init --scan --resume
```

### 实现设计

#### 1. 新增缓存管理模块（`internal/cache/checkpoint.go`）

```go
package cache

const CacheDir = ".kontext/.cache"

// Checkpoint 保存扫描流程的检查点信息
type Checkpoint struct {
    Version         string            `json:"version"`
    CreatedAt       time.Time         `json:"created_at"`
    ProjectHash     string            `json:"project_hash"`
    CompletedStages []int             `json:"completed_stages"`
    CurrentStage    int               `json:"current_stage"`

}

// SaveStageResult 保存某个阶段的结果
func SaveStageResult(stage int, data interface{}) error

// LoadStageResult 加载某个阶段的结果
func LoadStageResult(stage int, out interface{}) error

// IsCheckpointValid 检查缓存是否有效（项目结构未变化）
func IsCheckpointValid() (bool, *Checkpoint, error)

// ClearCache 清除所有缓存
func ClearCache() error
```

#### 2. 修改 `cmd/init.go` 的 runScanInit 流程

```go
func runScanInit() error {
    // 检查是否存在有效缓存
    valid, checkpoint, _ := cache.IsCheckpointValid()

    if valid && !freshFlag {
        if !resumeFlag {
            // 询问用户
            fmt.Printf("发现未完成的扫描任务（已完成阶段 1-%d），是否继续？[Y/n] ",
                checkpoint.CurrentStage-1)
            // ...
        }
        return runScanInitFromCheckpoint(checkpoint)
    }

    return runScanInitFresh()
}

func runScanInitFresh() error {
    // 阶段 1
    allFiles, err := fileutil.ScanDirectoryTree(...)
    if err != nil { return err }
    cache.SaveStageResult(1, allFiles)  // 保存检查点

    // 阶段 2
    analyzed, err := generator.AnalyzeProjectFiles(...)
    if err != nil { return err }
    cache.SaveStageResult(2, analyzed)  // 保存检查点

    // ... 依次类推
}

func runScanInitFromCheckpoint(cp *cache.Checkpoint) error {
    fmt.Printf("从阶段 %d 继续...\n\n", cp.CurrentStage)

    // 加载已完成阶段的结果
    var allFiles []string
    var analyzed *generator.AnalyzedFiles
    // ...

    cache.LoadStageResult(1, &allFiles)
    cache.LoadStageResult(2, &analyzed)
    // ...

    // 从中断的阶段继续
    switch cp.CurrentStage {
    case 6:
        return continueFromStage6(...)
    case 7:
        return continueFromStage7(...)
    // ...
    }
}
```

### 缓存过期策略

缓存在以下情况下自动失效：

1. **项目结构变化**：通过对目录树计算 hash，如果 hash 变化则缓存失效
2. **手动清除**：用户使用 `--fresh` 参数
3. **超时过期**：缓存超过 24 小时自动失效
4. **版本不兼容**：kontext 升级后缓存格式可能变化

## 优化三：单次 LLM 调用的容错（核心问题）

### 问题本质

当前阶段6的核心问题是：**一次 LLM 调用耗时 300-400 秒，期间任何网络抖动都会导致 EOF，全部白费**。

```
                        ┌─────────────────────────────────────┐
用户等待 384 秒 ──────────►│  一次超大 LLM 调用（生成4类物料）   │
                        └─────────────────────────────────────┘
                                        ↓
                              网络中断 → 全部丢失
```

即使有阶段间的检查点缓存，**阶段6内部那次漫长的 LLM 调用本身没有任何容错**。

### 解决方案：三管齐下

#### 方案 A：分步生成（最重要）

把一次 300+ 秒的大调用，拆成多次 10-30 秒的小调用：

```
当前：
┌──────────────────────────────────────────────────────────────┐
│ 一次调用：生成 MANIFEST + ARCH + CONV + 5个 CONTRACT        │
│ 耗时：300-400秒，中断概率高                                  │
└──────────────────────────────────────────────────────────────┘

优化后：
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ MANIFEST │→│   ARCH   │→│   CONV   │→│CONTRACT 1│→│CONTRACT 2│→ ...
│  ~15秒   │ │  ~10秒   │ │  ~10秒   │ │   ~8秒   │ │   ~8秒   │
└──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘
     ↓            ↓            ↓            ↓            ↓
   立即保存     立即保存     立即保存     立即保存     立即保存
```

**关键收益**：

- 单次调用耗时从 300秒 → 10-15秒，网络中断概率降低 95%
- 每完成一个文件立即写入磁盘，已完成的不丢失
- 中断后只需重试当前这一个文件，不是全部

#### 方案 B：流式输出（Stream）

即使单次调用中途断了，已经流式接收的部分内容不会丢失：

```go
// 当前：等待完整响应
resp, err := client.Chat(req)  // 等 300 秒，EOF 了什么都没有

// 优化：流式接收
stream, err := client.ChatStream(req)
var content strings.Builder
for chunk := range stream {
    content.WriteString(chunk)
    // 实时显示进度：已接收 1.2KB / 估计 3KB
}
// 即使 EOF，content 里已有部分内容
```

对于 YAML 生成，流式输出还有额外好处：

- 用户能实时看到生成进度（"正在生成 tech_stack 部分..."）
- 部分内容可能已经是完整可用的（比如 YAML 的前半部分）

#### 方案 C：自动重试 + 指数退避

网络错误时自动重试，而不是直接报错退出：

```go
func ChatWithRetry(client llm.Client, req *ChatRequest, maxRetries int) (*ChatResponse, error) {
    var lastErr error
    for i := 0; i < maxRetries; i++ {
        resp, err := client.Chat(req)
        if err == nil {
            return resp, nil
        }

        // 判断是否可重试的错误（网络错误、超时、5xx）
        if !isRetryableError(err) {
            return nil, err
        }

        lastErr = err
        backoff := time.Duration(math.Pow(2, float64(i))) * time.Second
        fmt.Printf("   ⚠ 网络错误，%v 后重试 (%d/%d)...\n", backoff, i+1, maxRetries)
        time.Sleep(backoff)
    }
    return nil, fmt.Errorf("重试 %d 次后仍然失败: %w", maxRetries, lastErr)
}

func isRetryableError(err error) bool {
    errStr := err.Error()
    return strings.Contains(errStr, "EOF") ||
           strings.Contains(errStr, "timeout") ||
           strings.Contains(errStr, "connection reset") ||
           strings.Contains(errStr, "503") ||
           strings.Contains(errStr, "502")
}
```

### 综合效果对比

| 场景     | 当前        | 分步生成     | +流式输出     | +自动重试 |
| -------- | ----------- | ------------ | ------------- | --------- |
| 正常完成 | 300秒一次性 | 分8次×15秒   | 同左+实时进度 | 同左      |
| 网络中断 | 全部丢失    | 丢失当前文件 | 部分内容可用  | 自动恢复  |
| 恢复代价 | 从阶段1重来 | 重试1个文件  | 可能0代价     | 自动处理  |

### 实现优先级

1. **P0（必做）**：分步生成 - 根本上降低单次调用的风险
2. **P1（推荐）**：自动重试 - 网络抖动自动恢复，用户无感
3. **P2（可选）**：流式输出 - 更好的用户体验和更强的容错

| **优化三** | 单次调用中断     | 自动重试 + 流式 | P1/P2   |
