# 优化模块契约生成流程

## 当前问题分析

### 问题 1：模块列表与 ARCHITECTURE_MAP 脱节

`extractModulesFromArch(archContent, allFiles)` 函数名写着"从 ARCHITECTURE_MAP 提取"，但实际完全没有使用 `archContent` 参数。模块列表是通过硬编码目录规则扫描得出的：

```go
// 当前逻辑：纯目录规则
internal/xxx/... → xxx
cmd/...          → cmd
pkg/xxx/...      → xxx
templates/...    → templates
```

**后果**：可能出现"ARCHITECTURE_MAP 里没有定义这个模块，但契约里生成了"的不一致。

### 问题 2：每个模块的 prompt 包含全部源码

当前 `userMsgGenerator` 的拼接方式：

```
baseUserMsg（全项目源码数据，可能几万 Token）
+ contractContext（MANIFEST + ARCHITECTURE 全文）
+ "请只为模块 xxx 生成..."
```

10 个模块 = 发送 10 份完整源码，大量重复 Token 浪费。

### 问题 3：模块间依赖关系不一致

每个模块的契约独立生成，没有全局约束。可能出现：

- 模块 A 的 `depends_on` 列出模块 B，但模块 B 的 `not_responsible_for` 里没有对应
- 循环依赖（A → B → C → A）

### 问题 4：失败模块没有二次重试机制

`GenerateModuleContracts` 完成后，失败的模块只打印错误日志，不会在外层做二次重试。虽然函数内部有单次重试（3 次），但如果 3 次全部失败，外层就直接放弃了。

### 问题 5：并行生成缺乏即时反馈

当前阶段 8 的用户体验：

```
📦 阶段 8/8：生成模块契约... (10 个模块并行)
   ⠙ 生成模块契约... (45秒)          ← 长时间只有一个转圈动画
   ✓ generator_CONTRACT.yaml          ← 全部完成后才一次性打印
   ✓ config_CONTRACT.yaml
   ...
```

应该在每个模块完成时立即打印，让用户看到实时进度。

### 问题 6：无流式输出支持

当前所有 LLM 调用都是等待完整响应后返回。对于长内容生成，用户会长时间看到转圈动画而无法了解进度。

### 改动 1（P0）：从 ARCHITECTURE_MAP 解析模块列表

#### 修改文件

- `cmd/init.go` — 重写 `extractModulesFromArch` 函数

#### 设计

```go
// 定义 ARCHITECTURE_MAP 的解析结构
type archLayer struct {
    Name     string   `yaml:"name"`
    Packages []string `yaml:"packages"`
}
type archMap struct {
    Layers []archLayer `yaml:"layers"`
}

func extractModulesFromArch(archContent string, allFiles []string) []string {
    // 优先从 ARCHITECTURE_MAP 解析
    var arch archMap
    if err := yaml.Unmarshal([]byte(archContent), &arch); err == nil && len(arch.Layers) > 0 {
        moduleSet := make(map[string]bool)
        for _, layer := range arch.Layers {
            for _, pkg := range layer.Packages {
                // "internal/config" → "config"
                // "cmd" → "cmd"
                // "templates" → "templates"
                name := normalizeModuleName(pkg)
                if name != "" {
                    moduleSet[name] = true
                }
            }
        }

        if len(moduleSet) > 0 {
            var modules []string
            for mod := range moduleSet {
                modules = append(modules, mod)
            }
            sort.Strings(modules)
            return modules
        }
    }

    // 解析失败时回退到目录规则扫描
    return fallbackExtractModules(allFiles)
}

// normalizeModuleName 将包路径规范化为模块名
// "internal/config" → "config"
// "cmd" → "cmd"
// "pkg/utils" → "utils"
func normalizeModuleName(pkg string) string {
    pkg = strings.TrimSpace(pkg)
    pkg = filepath.ToSlash(pkg)
    parts := strings.Split(pkg, "/")

    // 排除无意义的目录
    excluded := map[string]bool{"testdata": true, "vendor": true}

    // 取最后一级有意义的名称
    for i := len(parts) - 1; i >= 0; i-- {
        if parts[i] != "" && !excluded[parts[i]] {
            return parts[i]
        }
    }
    return ""
}

// fallbackExtractModules 是原来的目录扫描逻辑，作为回退方案
func fallbackExtractModules(allFiles []string) []string {
    // ... 原有的目录扫描逻辑
}
```

#### 新增依赖

- `cmd/init.go` 需要 import `"sort"` 和 `"go.yaml.in/yaml/v4"`

#### 修改文件

- `internal/generator/engine.go` — 新增 `FilterFilesByModule` 辅助函数

#### 设计

在 `internal/generator/engine.go` 新增：

```go
// FilterFilesByModule 筛选属于指定模块的文件
func FilterFilesByModule(files map[string]string, moduleName string) map[string]string {
    result := make(map[string]string)
    for path, content := range files {
        if BelongsToModule(path, moduleName) {
            result[path] = content
        }
    }
    return result
}

// BelongsToModule 判断文件路径是否属于指定模块
func BelongsToModule(filePath, moduleName string) bool {
    normalized := filepath.ToSlash(filePath)
    parts := strings.Split(normalized, "/")

    // cmd/xxx.go → 属于 cmd 模块
    if parts[0] == moduleName {
        return true
    }

    // internal/config/xxx.go → 属于 config 模块
    if len(parts) >= 2 && (parts[0] == "internal" || parts[0] == "pkg") {
        return parts[1] == moduleName
    }

    return false
}
```

在 `cmd/init.go` 中修改 `userMsgGenerator`：

```go
userMsgGenerator := func(moduleName string) (string, error) {
    // 筛选该模块的文件
    moduleKeyFiles := generator.FilterFilesByModule(keyFileContents, moduleName)
    moduleSummaries := generator.FilterFilesByModule(otherSummaries, moduleName)

    // 用精简数据构建用户消息
    moduleUserMsg, err := generator.RenderTemplate(templates.InitScanUser, map[string]interface{}{
        "DirectoryTree":      treeStr,              // 目录树保留（较小）
        "ConfigFiles":        configFiles,           // 配置文件保留
        "KeyFileContents":    moduleKeyFiles,        // 只包含该模块的源码
        "OtherFileSummaries": moduleSummaries,       // 只包含该模块的概要
    })
    if err != nil {
        return "", err
    }

    return moduleUserMsg + contractContext +
        fmt.Sprintf("\n\n请只为模块 `%s` 生成一个 CONTRACT.yaml 文件。不要生成其他模块或其他类型的文件。", moduleName), nil
}
```

#### 预期收益

- 每个模块的 prompt Token 量从 ~10000 降到 ~2000-3000
- LLM 更聚焦，契约质量更高
- 并发时 rate limit 压力更小

### 改动 3（P1）：失败模块二次重试

#### 修改文件

- `cmd/init.go` — 在阶段 8 的主逻辑中增加失败重试循环

#### 设计

```go
contracts, contractErrors := generator.GenerateModuleContracts(
    client, templates.InitScanContractSystem,
    modules, userMsgGenerator, 3, onProgress,
)
close(done5)

// 打印首轮结果...（原有逻辑）

// 对失败的模块进行二次重试
if len(contractErrors) > 0 {
    failedModules := extractFailedModuleNames(contractErrors)
    fmt.Printf("\n   ⚠ %d 个模块失败，正在重试...\n", len(failedModules))

    done6 := make(chan struct{})
    retryStart := time.Now()
    go spinnerAnimation(done6, retryStart, []string{"重试失败模块"})

    retryContracts, retryErrors := generator.GenerateModuleContracts(
        client, templates.InitScanContractSystem,
        failedModules, userMsgGenerator, 2, nil,
    )
    close(done6)

    // 合并重试成功的结果
    for mod, content := range retryContracts {
        contracts[mod] = content
    }

    // 校验并写入重试成功的文件
    for _, mod := range failedModules {
        if content, ok := retryContracts[mod]; ok {
            // ... 校验 + 写入 + 打印 ✓
        }
    }

    // 最终仍然失败的模块
    if len(retryErrors) > 0 {
        fmt.Printf("   ⚠ %d 个模块最终生成失败\n", len(retryErrors))
        for _, e := range retryErrors {
            fmt.Printf("      - %v\n", e)
        }
    }
}

// 辅助函数：从错误列表中提取失败的模块名
func extractFailedModuleNames(errors []error) []string {
    var names []string
    for _, e := range errors {
        // 错误格式: "模块 xxx: ..."
        msg := e.Error()
        if strings.HasPrefix(msg, "模块 ") {
            parts := strings.SplitN(msg[len("模块 "):], ":", 2)
            if len(parts) > 0 {
                names = append(names, strings.TrimSpace(parts[0]))
            }
        }
    }
    return names
}
```

**新增类型定义**（`internal/generator/types.go`）：

```go
// ModuleDependencyGraph 是模块间依赖关系图
type ModuleDependencyGraph struct {
    Modules []ModuleDep `json:"modules"`
}

type ModuleDep struct {
    Name      string   `json:"name"`       // 模块名
    Path      string   `json:"path"`       // 模块路径
    Purpose   string   `json:"purpose"`    // 一句话描述
    DependsOn []string `json:"depends_on"` // 依赖的模块名列表
}
```

**新增模板**（`templates/init_scan_depgraph_system.tmpl`）：

```
你是一个专业的软件架构师。

你将收到一个项目的源码信息和已生成的 ARCHITECTURE_MAP。
请分析各模块之间的依赖关系，生成模块依赖关系图。

你必须严格以 JSON 格式回复：
{
  "modules": [
    {
      "name": "模块名",
      "path": "模块路径",
      "purpose": "一句话描述模块职责",
      "depends_on": ["依赖的模块名1", "依赖的模块名2"]

  ]

要求：
- 依赖关系必须基于源码中实际的 import/调用关系
- 不允许出现循环依赖
- depends_on 中的模块名必须出现在 modules 列表中
```

```
     ↓
阶段 9/9：各模块契约（参考依赖图约束）
```


### 改动 5（P0）：并行模块契约即时反馈

#### 修改文件

- `cmd/init.go` — 启用 `onProgress` 回调，替换转圈动画

#### 设计

当前阶段 8 使用 `spinnerAnimation` 做转圈动画，所有模块完成后才打印结果。应改为利用已有的 `onProgress` 回调实现即时反馈：

```go
// 使用 mutex 保护控制台输出
var printMu sync.Mutex

onProgress := func(result generator.ModuleContractResult) {
    printMu.Lock()
    defer printMu.Unlock()

    if result.Error != nil {
        fmt.Printf("   ✗ %s_CONTRACT.yaml 失败: %v\n", result.ModuleName, result.Error)
    } else {
        fmt.Printf("   ✓ %s_CONTRACT.yaml (%.1f 秒)\n", result.ModuleName, result.Duration)
    }
}

// 不再使用 spinnerAnimation，直接调用
contracts, contractErrors := generator.GenerateModuleContracts(
    client,
    templates.InitScanContractSystem,
    modules,
    userMsgGenerator,
    3,
    onProgress,  // 传入即时回调
)
```

**效果对比**：

```
当前：
📦 阶段 8/8：生成模块契约... (10 个模块并行)
   ⠙ 生成模块契约... (45秒)         ← 45秒只有转圈
   ✓ generator_CONTRACT.yaml         ← 一次性全部输出
   ✓ config_CONTRACT.yaml
   ...

优化后：
📦 阶段 8/9：生成模块契约... (10 个模块并行)
   ✓ config_CONTRACT.yaml (5.2 秒)      ← 第 5 秒就看到
   ✓ fileutil_CONTRACT.yaml (6.1 秒)    ← 第 6 秒
   ✓ llm_CONTRACT.yaml (8.3 秒)         ← 实时出现
   ✓ generator_CONTRACT.yaml (9.7 秒)
   ...
```

注意：去掉 `spinnerAnimation` 后，需要确保即使并发输出也不会乱行。`printMu` 互斥锁保证了每条消息的原子性输出。

同时在 `GenerateModuleContracts` 内部也需要确保 `onProgress` 调用在 mutex 保护范围外（避免死锁），当前代码中 `onProgress` 的调用已经在 `mu.Lock()` 范围内——需要将 `onProgress` 移到锁外面：

```go
// engine.go 中 GenerateModuleContracts 的 goroutine 内
mu.Lock()
if err != nil {
    errors = append(errors, ...)
} else {
    results[moduleName] = contract.Content
}
progressResult := ModuleContractResult{...}
mu.Unlock()

// 在锁外面回调，避免死锁
if onProgress != nil {
    onProgress(progressResult)
}
```

### 改动 6（P2）：流式输出支持

#### 修改文件

- `internal/llm/types.go` — Client 接口新增 `ChatStream` 方法
- `internal/llm/openai.go` — 实现 `ChatStream`
- `internal/generator/engine.go` — `GenerateSingleYAML` 和 `GenerateModuleContract` 支持流式模式

#### 设计

**接口扩展**（`internal/llm/types.go`）：

```go
type Client interface {
    // 现有方法
    Generate(req *GenerateRequest) (*GenerateResponse, error)
    Chat(req *ChatRequest) (*ChatResponse, error)
    ChatStructured(req *ChatRequest, schemaName string, out any) (*ChatResponse, error)
    ListModels() ([]string, error)

    // 新增：流式输出
    ChatStream(req *ChatRequest, onChunk func(chunk string)) (*ChatResponse, error)
}
```

**OpenAI 实现**（`internal/llm/openai.go`）：

```go
func (c *openaiClient) ChatStream(req *ChatRequest, onChunk func(chunk string)) (*ChatResponse, error) {
    ctx, cancel := context.WithTimeout(context.Background(), c.cfg.GetTimeout())
    defer cancel()

    msgs := // ... 转换消息（同 createChatCompletion）

    stream := c.client.Chat.Completions.NewStreaming(ctx, openai.ChatCompletionNewParams{
        Model:    shared.ChatModel(c.cfg.Model),
        Messages: msgs,
    })

    var content strings.Builder
    for stream.Next() {
        event := stream.Current()
        for _, choice := range event.Choices {
            chunk := choice.Delta.Content
            if chunk != "" {
                content.WriteString(chunk)
                if onChunk != nil {
                    onChunk(chunk)
                }
            }
        }
    }

    if err := stream.Err(); err != nil {
        // 流中断但已有部分内容 → 返回部分内容 + 错误
        if content.Len() > 0 {
            return &ChatResponse{Content: content.String()}, err
        }
        return nil, fmt.Errorf("流式调用失败: %w", err)
    }

    return &ChatResponse{Content: content.String()}, nil
}
```

**在生成函数中使用**：

流式输出主要用于提升用户体验。对于 `GenerateSingleYAML`（阶段 6/7），可以在回退到 `Chat` 时改用 `ChatStream`，让用户看到已接收的字节数：

```go
// 回退路径中使用流式
var received int
resp, err := client.ChatStream(req, func(chunk string) {
    received += len(chunk)
    // 在 spinner 中显示进度：已接收 1.2KB
})
```

但流式输出与结构化输出（JSON Schema）不兼容，所以：
- **结构化输出（首选）**：不用流式，直接等完整结果
- **传统回退（Chat）**：改用流式，提升体验

#### 局限性

- OpenAI 的 `response_format: json_schema` 不支持 streaming（部分提供商支持）
- 流式收到的是纯文本 chunk，需要等全部收完后才能做 JSON 解析
- 主要收益是"用户能看到进度"而非"更快拿到结果"

#### 建议

流式输出改动涉及面广（接口变更、所有实现都要改），建议作为 P2 优先级，先完成 P0/P1 后再做。
