# Kontext Pack 改进方案（四项优先级）

---

## 一、Pack 流式输出 + 重试机制

### 1.1 问题现状

当前 `engine.Pack` 阶段 5 调用的是 `llm.Client.Generate()`，这是一个同步阻塞调用。对于 pack 场景，LLM 需要生成一份完整的 Markdown 文档（通常数千 token），等待时间可能在 30 秒到 2 分钟之间。这段时间里终端完全没有输出，用户无法判断是在工作还是卡死了。

同时，`Generate()` 没有走 `retry.go` 中已有的重试逻辑（`ChatWithRetry` / `ChatStreamWithRetry`），网络抖动或临时 5xx 会直接导致整个 pack 失败。

作为对比，`init --scan` 流程已经用了 `ChatStream` + `ChatWithRetry`，体验明显更好。

### 1.2 修改思路

#### 1.2.1 Engine 增加流式支持

在 `internal/packer/engine.go` 中：

- 将阶段 5 从 `e.llmClient.Generate(req)` 改为使用 `e.llmClient.ChatStream(chatReq, onChunk)`
- 需要将 `GenerateRequest`（system + user）转换为 `ChatRequest`（Messages 数组），这只是一个简单的格式转换：system prompt 作为 role=system 的 Message，user prompt 作为 role=user 的 Message
- `onChunk` 回调中，将每个 delta 片段实时打印到 stderr（用 `fmt.Fprint(os.Stderr, delta)`），这样不影响 stdout 的结构化输出
- 最终 `ChatStream` 返回的 `ChatResponse.Content` 就是完整的 Markdown 内容，直接传给阶段 6 保存

#### 1.2.2 增加重试包装

在 `internal/llm/retry.go` 中：

- 新增 `ChatStreamWithRetry` 函数，签名类似 `ChatWithRetry`，但额外接受 `onChunk func(string) error` 参数
- 重试逻辑与 `ChatWithRetry` 完全相同（指数退避、最大 30s、可重试错误判断复用 `isRetryableError`）
- 每次重试前通过 `onRetry` 回调通知用户（打印"正在重试第 N 次..."）
- 注意：重试时需要重新开始流式接收，之前接收到的部分内容作废。终端输出可以在重试前打印一个换行 + 提示信息来隔开

#### 1.2.3 Engine 中使用重试包装

在 `engine.go` 阶段 5 中：

- 调用 `llm.ChatStreamWithRetry(e.llmClient, chatReq, onChunk, 3, onRetry)`
- `onRetry` 回调中打印：`fmt.Fprintf(os.Stderr, "\n⚠ LLM 调用失败(%s)，%s 后重试第 %d 次...\n", err, backoff, attempt)`

#### 1.2.4 进度提示优化

在 `cmd/pack.go` 中：

- 阶段 1-4 在调用前分别打印简短的阶段提示，例如：
  - `[1/6] 加载 .kontext/ 配置...`
  - `[2/6] 扫描项目文件并匹配上下文...`
  - `[3/6] 构建提示词模板...`
  - `[4/6] 渲染提示词...`
  - `[5/6] 调用 LLM 生成 Prompt 文档...`（之后接流式输出）
  - `[6/6] 保存文件...`
- 这需要让 `Engine.Pack` 接受一个可选的 `onProgress func(stage int, msg string)` 回调，或者直接在 Engine 内部打印（更简单，但耦合 IO）
- 建议采用回调方式：给 `Engine` 加一个 `OnProgress func(stage, total int, msg string)` 字段，`cmd/pack.go` 中设置具体的打印行为

#### 1.2.5 涉及文件清单

| 文件 | 改动 |
|------|------|
| `internal/llm/retry.go` | 新增 `ChatStreamWithRetry` |
| `internal/packer/engine.go` | 阶段 5 改为流式 + 重试，增加 OnProgress 回调 |
| `cmd/pack.go` | 设置 OnProgress 回调实现终端打印 |

## 二、Pack 智能上下文筛选（LLM 二次筛选）

### 2.1 问题现状

当前 `collector.go` 的上下文收集完全基于关键词匹配：

1. `extractKeywords` 从任务描述中提取非停用词的关键词
2. `MatchFiles` 用这些关键词对文件路径做 `strings.Contains` 匹配
3. 如果匹配到 0 个文件，回退为所有 `.go` 文件
4. 硬上限 20 个文件，每个只读前 100 行
5. `MatchContracts` 用同样的关键词匹配模块契约

这种方式的问题：

- 关键词匹配是路径级的字符串包含，"implement user authentication" 只会匹配路径中含 "user" 或 "authentication" 的文件，可能遗漏核心业务逻辑文件
- 中文任务描述的关键词提取效果差（中文没有空格分词）
- 回退策略过于粗暴（所有 Go 文件）
- 每个文件只读前 100 行，可能完全读不到关键逻辑（大文件的核心函数在后半部分）
- 没有利用已有的 `ExtractFileSummary`（提取签名摘要）来做更精准的判断

### 2.2 修改思路

#### 2.2.1 整体策略：两轮筛选

将阶段 2（Discover Candidate Context）拆分为两步：

- **2a - 粗筛**：保留现有的关键词匹配逻辑，但扩大候选范围（从 20 个提升到 50 个），并且同时使用 `ExtractFileSummary` 获取每个候选文件的签名摘要（而非前 100 行原始代码）
- **2b - 精筛（LLM）**：将任务描述 + 候选文件列表（路径 + 签名摘要）发送给 LLM，由 LLM 返回结构化结果，指出哪些文件真正相关、相关程度（高/中/低）、以及每个文件中建议重点阅读的函数/区域

#### 2.2.2 粗筛改造（collector.go）

修改 `CollectContext` 函数：

- 新增常量 `maxCandidateFiles = 50`（粗筛阶段的上限，比最终给 LLM 的 20 个要大）
- 关键词匹配仍然保留，但增加一个辅助匹配维度：用 `ExtractFileSummary` 对所有代码文件提取签名摘要，然后在签名文本中也做关键词匹配（而不仅仅是文件路径）
- 新增 `CandidateContext.FileSummaries map[string]string` 字段，保存粗筛阶段每个候选文件的签名摘要
- 粗筛结果为：候选文件路径列表 + 每个文件的签名摘要

#### 2.2.3 精筛（新增 refiner.go）

在 `internal/packer/` 下新增 `refiner.go`：

- 定义 `RefineResult` 结构体：

  ```
  type FileRelevance struct {
      Path       string   // 文件路径
      Relevance  string   // "high" / "medium" / "low"
      Reason     string   // 为什么相关
      FocusAreas []string // 建议重点阅读的函数/区域名
  }
  type RefineResult struct {
      RelevantFiles []FileRelevance
  }
  ```

- 定义 `RefineContext(client llm.Client, task string, candidates []CandidateFile, contracts []schema.ModuleContract) (*RefineResult, error)` 函数
- 构造发给 LLM 的输入：
  - system prompt：「你是代码分析专家，根据任务描述判断哪些文件与任务相关」
  - user prompt：任务描述 + 候选文件列表（每个文件给出路径和签名摘要，不给全文代码）+ 模块契约列表
- 使用 `ChatStructuredWithRetry` 调用 LLM，要求输出 `RefineResult` 的 JSON
- 返回值中只保留 relevance 为 high 和 medium 的文件

#### 2.2.4 精筛后的代码读取策略

在 `collector.go` 中根据精筛结果决定如何读取代码：

- **high 相关度的文件**：读取完整内容（不限行数，或上限提高到 500 行）；如果 LLM 返回了 `FocusAreas`，优先定位这些函数并读取完整函数体
- **medium 相关度的文件**：只读取签名摘要（`ExtractFileSummary` 结果），不读全文
- **low / 未提及的文件**：不包含在最终上下文中

这样做的好处是：高相关文件给足代码上下文，低相关文件不浪费 token，整体质量远超"每个文件前 100 行"。

#### 2.2.5 新增模板文件

在 `templates/` 下新增：

- `pack_refine_system.tmpl`：精筛 LLM 的 system prompt
- `pack_refine_user.tmpl`：精筛 LLM 的 user prompt 模板（接收任务描述、候选文件列表、模块契约）

同时在 `templates/templates.go` 中用 `//go:embed` 注册这两个新模板。

#### 2.2.6 Engine 流程调整

`engine.go` 的 `Pack` 方法需要改为 7 阶段：

1. 加载 Bundle
2. 粗筛候选上下文（扩大范围，收集签名摘要）
3. **LLM 精筛**（新增阶段，调用 refiner）
4. 根据精筛结果读取代码（替换原来的一刀切读取）
5. 构建模板数据
6. 渲染提示词 + 调用 LLM 生成 Prompt 文档
7. 保存

注意：Engine 现在在两个地方调用 LLM（阶段 3 精筛 + 阶段 6 生成）。精筛使用 `ChatStructured`（结构化输出，速度快，token 少），生成使用 `ChatStream`（流式输出，内容长）。

#### 2.2.7 可选开关

在 `cmd/pack.go` 中增加 `--no-refine` 标志，允许用户跳过 LLM 精筛步骤（退回到纯关键词匹配模式），用于节省 API 调用或调试。

#### 2.2.8 涉及文件清单

| 文件 | 改动 |
|------|------|
| `internal/packer/collector.go` | 扩大候选范围，增加签名摘要收集 |
| `internal/packer/refiner.go` | **新增**，LLM 精筛逻辑 |
| `internal/packer/engine.go` | 拆分为 7 阶段，插入精筛步骤 |
| `internal/packer/builder.go` | 适配新的分级代码读取结果 |
| `internal/packer/prompt_template.go` | TemplateData 可能需要增加字段 |
| `templates/pack_refine_system.tmpl` | **新增** |
| `templates/pack_refine_user.tmpl` | **新增** |
| `templates/templates.go` | 注册新模板 |
| `cmd/pack.go` | 增加 `--no-refine` 标志 |

## 三、Pack --from-file 支持

### 3.1 问题现状

当前 `pack` 命令只接受一个字符串参数作为任务描述：

```bash
kontext pack "实现用户认证功能"
```

对于复杂任务，一句话描述不够用。用户可能想用一个 Markdown 文件详细描述需求，包括背景说明、验收标准、约束条件等。当前必须把所有内容塞进一个命令行字符串参数，很不方便。

### 3.2 修改思路

#### 3.2.1 新增 --from-file 标志

在 `cmd/pack.go` 中：

- 新增 `--from-file` / `-f` 标志，类型为 string，接受一个文件路径
- 修改参数校验逻辑：
  - 如果提供了 `--from-file`，则从文件读取任务描述，此时位置参数变为可选（`Args` 改为 `cobra.MaximumNArgs(1)`）
  - 如果同时提供了 `--from-file` 和位置参数，报错提示二者互斥
  - 如果都没提供，报错提示需要任务描述

#### 3.2.2 文件读取逻辑

在 `cmd/pack.go` 的 `RunE` 中：

- 当 `--from-file` 被设置时，调用 `fileutil.ReadFile(fromFilePath)` 读取文件内容
- 对读取的内容做基本清理：去除首尾空白、去除可能的 BOM 标记
- 读取到的完整文本作为 `task` 传给 `engine.Pack(task)`

#### 3.2.3 对 Prompt 模板的影响

当前 `llm_user_prompt.tmpl` 中 `Task` 字段只放了一句话。当 task 来自文件时，可能是几百字的详细描述。需要验证：

- user prompt 模板中 `{{.Task}}` 位置放长文本是否合理 — 是合理的，因为这个字段本身就是"任务描述"，长短不影响模板结构
- LLM system prompt 中已经要求输出固定结构的 Markdown，长任务描述只会让 LLM 理解得更准确

#### 3.2.4 文件名生成适配

当前 `promptdoc.GenerateFilename(task)` 会把整个 task 做 slug 处理。如果 task 是一篇长文，slug 可能过长。当前已有 50 字符截断逻辑，但可以做更好的处理：

- 如果 task 超过 200 字符，只取前 200 字符做 slug（截断前先找到最近的词边界）
- 或者当使用 `--from-file` 时，用文件名作为 slug 的一部分（例如 `20260320-143000_from-requirements.md`）

建议方案：在 `promptdoc.GenerateFilename` 增加可选的 `hint` 参数。`cmd/pack.go` 中如果是 `--from-file` 模式，传入源文件名作为 hint；否则传空字符串走原有逻辑。

#### 3.2.5 支持 stdin 输入（可选增强）

除了文件，还可以支持从 stdin 读取：

- 当 `--from-file` 设置为 `-` 时，从 stdin 读取
- 这允许管道用法：`cat requirements.md | kontext pack --from-file -`
- 实现方式：判断 fromFilePath 是否为 `-`，是则读 `os.Stdin`

这个作为可选增强，不在第一版要求内。

#### 3.2.6 涉及文件清单

| 文件 | 改动 |
|------|------|
| `cmd/pack.go` | 新增 `--from-file` 标志，修改参数校验和读取逻辑 |
| `internal/promptdoc/writer.go` | `GenerateFilename` 增加 hint 参数适配长任务 |

## 四、Kontext Update 命令

### 4.1 问题现状

`.kontext/` 目录中的物料（PROJECT_MANIFEST.yaml、ARCHITECTURE_MAP.yaml、CONVENTIONS.yaml、module_contracts/*.yaml）是在 `init` 时生成的，之后不会自动更新。随着项目演进，代码结构变化、新模块增加、旧模块重构，这些物料会逐渐过时。

目前用户只能重新运行 `init --scan` 来全量重新生成，但这样会覆盖掉用户手动调整过的内容，而且耗时较长（需要多次 LLM 调用）。

### 4.2 修改思路

#### 4.2.1 命令设计

```bash
kontext update                    # 检测变更并更新所有过时的物料
kontext update --dry-run          # 只检测变更，不实际修改，打印变更报告
kontext update --file manifest    # 只更新指定物料文件
kontext update --file contracts   # 只更新模块契约
kontext update --since <commit>   # 只分析指定 commit 之后的变更
```

#### 4.2.2 变更检测策略

**核心思路**：对比"当前代码实际状态"与".kontext/ 物料中描述的状态"，找出不一致的地方。

变更检测分三层：

**第一层：目录结构变更检测**

- 扫描当前项目目录结构（复用 `fileutil.ScanDirectoryTree`）
- 从 `ARCHITECTURE_MAP.yaml` 中提取已记录的目录/包列表
- 对比找出：新增的目录/包（存在于代码但不在 ARCHITECTURE_MAP 中）、已删除的目录/包（在 ARCHITECTURE_MAP 中但代码中已不存在）
- 输出：`[]DirectoryChange{Path, Type("added"/"removed"/"renamed")}`

**第二层：模块契约变更检测**

- 扫描 `module_contracts/` 下已有的契约文件，提取模块名列表
- 扫描代码目录中的实际包列表
- 对比找出：新增模块（代码中有但没有契约文件）、已删除模块（有契约文件但代码中不存在）
- 对于已有契约的模块，用 `ExtractFileSummary` 提取当前代码的函数签名列表，与契约中的 `owns` 字段做粗略对比，检测是否有大幅偏移（新增了很多未记录的导出函数、或 owns 中的函数已不存在）
- 输出：`[]ContractChange{Module, Type("new_module"/"deleted_module"/"stale_contract"), Details}`

**第三层：Git diff 辅助检测（可选，当 --since 提供时）**

- 调用 `git diff --name-only <since>..HEAD` 获取变更文件列表
- 分析变更文件归属的模块，标记这些模块的契约可能需要更新
- 这一层是辅助信息，不是必须的

#### 4.2.3 新增 internal/updater/ 包

创建 `internal/updater/` 目录，包含：

- **detector.go**：变更检测逻辑
  - `DetectChanges(kontextDir, projectDir string) (*ChangeReport, error)` — 执行上述三层检测
  - `ChangeReport` 结构体包含：目录变更列表、契约变更列表、整体变更摘要

- **planner.go**：更新计划生成
  - 根据 `ChangeReport`，决定哪些物料文件需要更新
  - 生成 `UpdatePlan`：`[]UpdateAction{Target("manifest"/"architecture"/"contract:xxx"), Reason, Priority}`
  - 更新优先级：ARCHITECTURE_MAP（目录结构变化时）> module_contracts（模块级变化时）> PROJECT_MANIFEST（通常不需要更新，除非技术栈变化）> CONVENTIONS（基本不会自动更新）

- **executor.go**：执行更新
  - 对于每个 `UpdateAction`，构造合适的 LLM 请求：
    - 将「当前物料内容 + 检测到的变更 + 当前代码签名摘要」发送给 LLM
    - 要求 LLM 输出更新后的 YAML 内容
    - 注意：不是从零生成，而是在现有内容基础上增量修改，这样保留用户手动调整过的部分
  - 使用 `ChatStructured` + 重试，确保输出是合法 YAML
  - 更新前先备份原文件到 `.kontext/.backup/`（带时间戳）
  - 写入更新后的文件
  - 调用 `schema.ValidateBundle` 验证更新后的物料是否合法

#### 4.2.4 LLM Prompt 设计要点

更新场景的 LLM prompt 与 init 的 prompt 有本质区别：

- init 是从零生成，prompt 重点是"理解项目并创建物料"
- update 是增量修改，prompt 重点是"理解变更并最小化修改现有物料"

需要新增的模板文件：

- `templates/update_system.tmpl`：system prompt，强调"在现有内容基础上做最小修改"
- `templates/update_architecture.tmpl`：更新 ARCHITECTURE_MAP 的 user prompt，输入是「当前 YAML + 目录变更列表 + 当前目录结构」
- `templates/update_contract.tmpl`：更新单个模块契约的 user prompt，输入是「当前契约 YAML + 该模块当前代码签名摘要 + 检测到的变更」
- `templates/update_manifest.tmpl`：更新 PROJECT_MANIFEST 的 user prompt（轻量，主要更新技术栈和命令列表）

#### 4.2.5 --dry-run 模式

`--dry-run` 是 update 命令的重要模式：

- 只执行变更检测（detector.go），不调用 LLM，不修改文件
- 输出一份人类可读的变更报告，例如：

  ```
  === Kontext 物料变更检测报告 ===

  [目录结构变更]
    + internal/updater/     (新增目录，ARCHITECTURE_MAP 中未记录)
    - internal/deprecated/  (已删除目录，ARCHITECTURE_MAP 中仍存在)

  [模块契约变更]
    + updater  (新模块，缺少契约文件)
    ~ packer   (契约可能过时：新增 3 个未记录的导出函数)

  [建议更新]
    1. ARCHITECTURE_MAP.yaml  — 2 处目录变更
    2. module_contracts/updater_CONTRACT.yaml — 新模块需要创建
    3. module_contracts/packer_CONTRACT.yaml — 建议更新

  运行 `kontext update` 执行更新。
  ```

这让用户在实际修改前先了解会发生什么。

#### 4.2.6 交互确认

非 dry-run 模式下，在每个文件更新前打印变更摘要并询问用户确认：

- 显示"即将更新 ARCHITECTURE_MAP.yaml，原因：检测到 2 处目录变更"
- 用户输入 y/n 决定是否继续
- 加 `--yes` / `-y` 标志跳过确认（自动化场景使用）

#### 4.2.7 备份策略

- 每次 update 前，将被修改的文件备份到 `.kontext/.backup/<timestamp>/`
- 例如 `.kontext/.backup/20260320-143000/ARCHITECTURE_MAP.yaml`
- 在 `.gitignore` 中建议用户忽略 `.kontext/.backup/`（或在 init 生成 .gitignore 建议时加入）
- 保留最近 5 次备份，自动清理更早的

#### 4.2.8 涉及文件清单

| 文件 | 改动 |
|------|------|
| `cmd/update.go` | **新增**，cobra 命令定义 |
| `cmd/root.go` | 注册 update 子命令 |
| `internal/updater/detector.go` | **新增**，变更检测 |
| `internal/updater/planner.go` | **新增**，更新计划 |
| `internal/updater/executor.go` | **新增**，执行更新 |

## 实施顺序与依赖关系

```
第一项（流式+重试）─── 无依赖，可立即开始，工作量最小
       │
       ▼
第二项（智能筛选）─── 依赖第一项（Engine 结构调整后再拆阶段更容易）
       │
       ▼
第三项（--from-file）── 无强依赖，可与第二项并行，但建议在第二项之后（因为长任务描述配合智能筛选效果更佳）
       │
       ▼
第四项（update）────── 独立新功能，工作量最大，建议最后做
```

预计总工作量：第一项约占 15%，第二项约占 30%，第三项约占 10%，第四项约占 45%。
