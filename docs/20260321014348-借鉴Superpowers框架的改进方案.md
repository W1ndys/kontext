# 借鉴 Superpowers 框架的改进方案

---

## 背景

[Superpowers](https://github.com/obra/superpowers) 是一个面向 AI 编码代理（Claude Code、Cursor、Codex、Gemini CLI、OpenCode）的 **行为技能框架**，由 Jesse Vincent 创建，目前 GitHub 星标超过 10 万。它的核心理念是：不让 AI "随便写代码"，而是通过一套**结构化的行为规范（Skills）**，将 AI 编码代理从临时代码生成器升级为遵循严格软件工程实践的开发者。

Kontext 和 Superpowers 解决的是互补问题：

| 维度     | Kontext                           | Superpowers                           |
| -------- | --------------------------------- | ------------------------------------- |
| 核心问题 | AI **不了解**项目上下文           | AI **不遵守**开发流程                 |
| 输出物   | `.kontext/` YAML 制品（知识编译） | `skills/` SKILL.md（行为编译）        |
| 技术栈   | Go + Cobra + Bubbletea + LLM      | Shell + JS + Markdown（几乎纯提示词） |
| 定位     | Context Compiler                  | Behavior Compiler                     |

本文档详细梳理 Superpowers 的架构与实现思路，并提炼出可应用于 Kontext 的 7 项改进方案。

---

## 第一部分：Superpowers 架构与实现思路详解

### 1.1 总体架构 — 三层插件体系

Superpowers 不是一个传统意义上的软件应用，而是一个**提示词工程框架**。其架构分为三层：

```
┌─────────────────────────────────────────────────────────────────┐
│  第一层：平台集成层（Platform Integration Layer）                  │
│                                                                  │
│  .claude-plugin/plugin.json   — Claude Code 插件清单             │
│  .cursor-plugin/plugin.json   — Cursor 插件清单                  │
│  gemini-extension.json        — Gemini CLI 扩展清单              │
│  .codex/INSTALL.md            — Codex 手动安装指引               │
│  .opencode/INSTALL.md         — OpenCode 手动安装指引            │
│                                                                  │
│  作用：让每个平台能"发现"和"加载"这个框架                         │
├─────────────────────────────────────────────────────────────────┤
│  第二层：钩子与调度层（Hook & Dispatch Layer）                     │
│                                                                  │
│  hooks/session-start          — 会话启动钩子（Bash 脚本）        │
│  hooks/hooks.json             — Claude Code 钩子配置             │
│  hooks/hooks-cursor.json      — Cursor 钩子配置                  │
│  hooks/run-hook.cmd           — Windows 兼容批处理               │
│  commands/                    — 斜杠命令（已废弃，迁移到技能）    │
│                                                                  │
│  作用：在 AI 会话生命周期的关键时刻注入行为指令                    │
├─────────────────────────────────────────────────────────────────┤
│  第三层：技能层（Skills Layer）                                    │
│                                                                  │
│  skills/using-superpowers/SKILL.md    — 元技能（技能调度器）      │
│  skills/brainstorming/SKILL.md        — 头脑风暴                 │
│  skills/writing-plans/SKILL.md        — 编写计划                 │
│  skills/executing-plans/SKILL.md      — 执行计划                 │
│  skills/subagent-driven-development/  — 子代理驱动开发（含4个文件）│
│  skills/test-driven-development/      — 测试驱动开发（含2个文件） │
│  skills/systematic-debugging/         — 系统化调试（含4个文件）   │
│  skills/verification-before-completion/— 完成前验证              │
│  skills/writing-skills/               — 编写技能（元技能，含7个文件）│
│  skills/dispatching-parallel-agents/  — 并行代理调度             │
│  skills/requesting-code-review/       — 请求代码审查             │
│  skills/receiving-code-review/        — 接收代码审查             │
│  skills/using-git-worktrees/          — Git Worktree 使用        │
│  skills/finishing-a-development-branch/— 完成开发分支            │
│                                                                  │
│  作用：14 个行为技能，定义 AI 在各场景下的具体工作流程             │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 会话引导机制 — Session Hook

Superpowers 的核心引导发生在 AI 会话启动的瞬间。`hooks/session-start` 是一个 Bash 脚本，执行以下关键步骤：

```
用户打开 AI 会话
        │
        ▼
hooks/session-start 被平台自动调用
        │
        ├── 检测运行环境（CLAUDE_PLUGIN_ROOT vs CURSOR_PLUGIN_ROOT）
        │
        ├── 读取 skills/using-superpowers/SKILL.md 全文
        │
        ├── 将内容 JSON 转义（处理换行、引号等）
        │
        ├── 输出为 JSON: {"additionalContext": "...转义后的SKILL.md全文..."}
        │
        ├── 检测是否存在旧版技能目录（legacy migration 警告）
        │
        └── AI 代理现在"知道"技能系统的存在，可以按需调用任何技能
```

技术细节：脚本使用 `printf` 而非 heredoc 来构造 JSON 输出，因为 Bash 5.3+ 的 heredoc 在内容超过 ~512 字节时会出现变量展开挂起的 bug（记录在 issue #571）。

### 1.3 技能设计模式 — SKILL.md 的结构

每个技能文件遵循统一格式：

```yaml
---
name: test-driven-development
description: |
  When writing code that needs tests, or when tests need to be written
  for existing code, or when fixing bugs that should have test coverage...
---

# Test-Driven Development

## When This Skill Applies
[触发条件的详细描述]

## The Iron Law
[核心不可违反的规则]

## Workflow
[具体步骤流程，通常含 DOT 语言流程图]

## Rationalization Table
| Excuse | Why It's Wrong |
|--------|----------------|
| "这个太简单不需要测试" | "简单的代码也会有 bug..." |
| ... | ... |

## Red Flags
[AI 自检清单]
```

**CSO（Claude Search Optimization）设计原则**：`description` 字段的措辞经过精心设计，目的是告诉 AI **"什么时候应该读取这个技能"**，而不是概括技能内容。如果 description 泄露了工作流摘要（例如"任务之间做代码审查"），AI 会直接按摘要行事而跳过阅读完整内容，导致只做一轮审查而非规定的两轮。

### 1.4 子代理驱动开发 — 最复杂的技能

`subagent-driven-development/` 是 Superpowers 的旗舰技能，包含 4 个文件：

```
subagent-driven-development/
├── SKILL.md                      — 控制器代理的行为规范
├── implementer-prompt.md         — 实现者子代理的提示词模板
├── spec-reviewer-prompt.md       — 规格审查员子代理的提示词模板
└── code-quality-reviewer-prompt.md — 代码质量审查员子代理的提示词模板
```

**完整工作流程：**

```
控制器代理读取计划
        │
        ▼
    ┌─────────────────────────────────────────────────┐
    │  对计划中的每个任务，循环执行：                      │
    │                                                   │
    │  1. 准备上下文包（任务全文，不是引用）                │
    │     ├── 完整任务描述                               │
    │     ├── 相关文件路径                               │
    │     ├── 技术约束和上下文                           │
    │     └── 模型级别建议                               │
    │                                                   │
    │  2. 派发实现者子代理                               │
    │     └── 子代理返回四种状态之一：                    │
    │         ├── DONE → 继续审查                        │
    │         ├── DONE_WITH_CONCERNS → 阅读关注点后审查    │
    │         ├── NEEDS_CONTEXT → 提供信息，重新派发       │
    │         └── BLOCKED → 评估阻塞，可能升级模型        │
    │                                                   │
    │  3. 派发规格合规审查员                              │
    │     └── 关键设计："实现者完成得异常迅速，              │
    │         他们的报告可能不完整。不要信任报告。"          │
    │     └── 审查员独立检查实现是否匹配规格               │
    │                                                   │
    │  4. 派发代码质量审查员                              │
    │     └── 检查：代码整洁度、可维护性、测试覆盖         │
    │                                                   │
    │  5. 如果任一审查未通过 → 返回步骤 2 重新实现         │
    │     如果两轮审查都通过 → 进入下一个任务               │
    └─────────────────────────────────────────────────┘
```

**模型成本优化策略：**

| 任务类型                     | 推荐模型级别  | 理由           |
| ---------------------------- | ------------- | -------------- |
| 机械性任务（孤立、规格明确） | 便宜/快速模型 | 不需要深度推理 |
| 集成性任务（多文件交互）     | 标准模型      | 需要理解上下文 |
| 架构/审查任务                | 最强模型      | 需要全局判断力 |

### 1.5 反合理化工程 — Superpowers 最独特的创新

Superpowers 发现了一个关键问题：AI 代理**非常擅长找借口跳过规则**。它会说"这个情况很简单不需要测试"、"先写代码再补测试更高效"、"这次赶时间先跳过审查"。

应对策略形成了一套系统化的**反合理化工程**：

#### 1.5.1 合理化表（Rationalization Table）

每个关键技能都包含一张表，左列是 AI 可能找的借口，右列是为什么这个借口不成立：

```markdown
| AI 的借口                        | 为什么不成立                                       |
| -------------------------------- | -------------------------------------------------- |
| "这段代码太简单了，不需要测试"   | "简单的代码也可能有边界条件 bug，而且测试也是文档" |
| "先写代码再补测试更高效"         | "这就是为什么你总是'忘记'补测试的原因"             |
| "时间紧迫，这次跳过审查"         | "审查发现的 bug 比后期调试节省的时间多得多"        |
| "我已经仔细检查过了，不需要验证" | "你的'仔细检查'和实际运行验证是两回事"             |
```

#### 1.5.2 红旗清单（Red Flags）

在 `using-superpowers` 元技能中，定义了 12 种"危险信号"：

> 如果你发现自己在想以下任何一条，**立即停下来并调用相关技能**：
>
> 1. "这个改动太小了，不需要走流程"
> 2. "我很确定这样是对的，不需要验证"
> 3. "用户没要求测试，所以不用写"
> 4. "先快速搞定，之后再完善"
>    ...共 12 条

#### 1.5.3 语言升级（Escalation of Language）

技能中使用越来越强烈的措辞来强调关键规则：

- 普通规则：`You should...`
- 重要规则：`You MUST...`、`EXTREMELY_IMPORTANT`
- 核心规则：`The Iron Law`、`HARD-GATE`、`NON-NEGOTIABLE`
- 终极强调：`"Violating the letter of the rules IS violating the spirit of the rules"`

#### 1.5.4 说服原理的应用

`writing-skills` 元技能甚至引用了**Cialdini 的说服原理**来指导如何编写防弹技能：

- **承诺与一致性**：让 AI 在早期承诺遵守规则
- **社会认同**：引用"优秀工程师都这样做"
- **权威性**：引用最佳实践和行业标准

### 1.6 TDD 应用于提示词本身

`writing-skills` 技能定义了一套**对提示词本身做 TDD** 的方法论：

```
RED 阶段（发现失败）：
├── 1. 选择一个压力场景（AI 可能偷懒的典型情况）
├── 2. 派发子代理，不给它任何技能指导
├── 3. 观察子代理的行为 → 记录所有失败模式
│       例如：跳过测试、合并不相关的步骤、不做审查
└── 4. 编写失败测试用例文档

GREEN 阶段（修复失败）：
├── 1. 编写最小化的 SKILL.md 来解决观察到的失败
├── 2. 重新派发子代理，这次带上技能
├── 3. 验证之前的失败模式是否被消除
└── 4. 如果没有 → 回到 RED 阶段找出漏洞

REFACTOR 阶段（巩固）：
├── 1. 用新的压力场景重新测试
├── 2. 发现新的合理化模式
├── 3. 在 Rationalization Table 中增加对应条目
└── 4. 重复直到 3 轮连续测试都通过
```

### 1.7 头脑风暴技能 — 苏格拉底式设计对话

`brainstorming/` 技能实现了一个完整的需求提取流程：

```
用户提出模糊需求："我想加个登录功能"
        │
        ▼
AI 进入苏格拉底式对话模式
├── 不立即给出方案
├── 通过追问理解真实需求：
│   "目标用户是谁？需要支持第三方登录吗？
│    对安全等级有什么要求？需要双因素认证吗？"
│
├── [可选] 启动可视化伴侣（浏览器端）
│   └── 实时渲染 UI 草图和流程图帮助对齐理解
│
├── 生成设计规格文档（Spec Document）
│
├── 派发规格审查员子代理
│   └── 独立审查规格的完整性和一致性
│   └── 最多 3 轮修改后交给人类确认
│
└── 输出：完整、明确、可执行的设计规格
```

### 1.8 指令优先级层次

Superpowers 明确定义了三级优先级，确保用户始终拥有最终控制权：

```
优先级 1（最高）：用户的显式指令（CLAUDE.md、GEMINI.md 等）
优先级 2（中）：  Superpowers 技能规范
优先级 3（最低）：AI 默认系统提示词
```

这意味着用户可以通过在 `CLAUDE.md` 中写 `"不要使用 TDD"` 来覆盖 Superpowers 的 TDD 强制规则。

## 第二部分：Kontext 可借鉴的 7 项改进方案

### 改进一：多平台输出适配层

**优先级：🔴 高（直接扩大用户群）**

#### 问题现状

当前 `kontext pack` 生成的 prompt 文档是通用 Markdown 格式，保存在 `.kontext/prompts/` 中。用户需要手动将内容复制粘贴到对应的 AI 工具中，摩擦成本高。

#### 借鉴思路

Superpowers 通过平台适配层（`.claude-plugin/`、`.cursor-plugin/`、`gemini-extension.json`）实现"一套技能，多平台消费"。我们可以对 `pack` 的输出做类似的适配。

#### 实现方案

**1. 新增 `--format` 参数，支持多种输出格式**

```bash
kontext pack "实现用户登录" --format cursor    # 输出 .cursorrules 格式
kontext pack "实现用户登录" --format claude     # 输出 CLAUDE.md 格式
kontext pack "实现用户登录" --format copilot    # 输出 .github/copilot-instructions.md 格式
kontext pack "实现用户登录" --format raw        # 默认，当前的通用 Markdown
```

**2. 在 `templates/` 中新增平台模板文件**

```
templates/
├── pack_output_raw.tmpl          # 当前的通用格式（默认）
├── pack_output_cursor.tmpl       # Cursor Rules 格式
├── pack_output_claude.tmpl       # CLAUDE.md 格式
└── pack_output_copilot.tmpl      # Copilot Instructions 格式
```

**3. 在 `internal/packer/` 中增加格式化层**

```go
// internal/packer/formatter.go

type OutputFormat string

const (
    FormatRaw     OutputFormat = "raw"
    FormatCursor  OutputFormat = "cursor"
    FormatClaude  OutputFormat = "claude"
    FormatCopilot OutputFormat = "copilot"
)

type Formatter struct {
    format OutputFormat
}

func (f *Formatter) Format(data *TemplateData) (string, error) {
    tmplName := fmt.Sprintf("pack_output_%s.tmpl", f.format)
    return templates.Render(tmplName, data)
}

func (f *Formatter) OutputPath(baseDir string) string {
    switch f.format {
    case FormatCursor:
        return filepath.Join(baseDir, ".cursorrules")
    case FormatClaude:
        return filepath.Join(baseDir, "CLAUDE.md")
    case FormatCopilot:
        return filepath.Join(baseDir, ".github", "copilot-instructions.md")
    default:
        return "" // 使用默认的 .kontext/prompts/ 路径
    }
}
```

**4. `cmd/pack.go` 增加 `--format` flag**

```go
packCmd.Flags().StringVarP(&formatFlag, "format", "f", "raw",
    "输出格式: raw, cursor, claude, copilot")
```

**5. 各平台模板差异点**

| 平台                  | 格式要求                                                     |
| --------------------- | ------------------------------------------------------------ |
| Cursor (.cursorrules) | 纯指令式描述，无需 Markdown 标题层级，直接 "You are..." 开头 |
| Claude (CLAUDE.md)    | Markdown 格式，支持层级标题，可包含代码块和规则              |
| Copilot               | Markdown 格式，偏向简洁的指令列表                            |
| Raw                   | 当前的完整 Markdown 文档（系统提示 + 用户提示）              |

#### 涉及文件

- 新增：`internal/packer/formatter.go`
- 新增：`templates/pack_output_cursor.tmpl`、`templates/pack_output_claude.tmpl`、`templates/pack_output_copilot.tmpl`
- 修改：`cmd/pack.go`（增加 `--format` 参数）
- 修改：`internal/packer/engine.go`（在阶段 6 前插入格式化步骤）
- 修改：`templates/templates.go`（嵌入新模板）

### 改进二：LLM 提示词反合理化约束

**优先级：🔴 高（提升生成质量，几乎零代码成本）**

#### 问题现状

当前 `init --scan` 和 `pack` 的 LLM 提示词模板（`templates/*.tmpl`）以正向指令为主，告诉 LLM "做什么"，但没有告诉它"不要偷哪些懒"。在实际使用中，LLM 经常出现以下偷懒行为：

- **合并不相关模块**：把 `internal/llm` 和 `internal/config` 写进同一个 contract（"它们都是基础设施"）
- **跳过小文件**：忽略只有几十行但功能关键的文件（"文件太短不重要"）
- **过度泛化描述**：用"处理各种相关逻辑"代替精确描述（"避免遗漏"）
- **遗漏接口细节**：只列出函数名而省略参数和返回值（"节省空间"）

#### 借鉴思路

直接应用 Superpowers 的 Rationalization Table 模式，在关键提示词模板中增加"反合理化"约束段。

#### 实现方案

**1. 在 `init_scan_*.tmpl` 相关模板中增加反合理化段**

以模块契约生成模板为例，在 prompt 末尾追加：

```markdown
## ⚠ 常见错误 — 你可能会犯但必须避免的

| 你可能会想                              | 为什么这是错的                                           |
| --------------------------------------- | -------------------------------------------------------- |
| "这两个包功能相似，合并成一个 contract" | 每个 Go package 必须有独立的 contract，无论多小          |
| "这个文件只有30行，不重要"              | 小文件往往定义关键接口，如 types.go 和 interface 定义    |
| "列出函数名就够了，签名太冗长"          | 没有参数和返回值的函数列表毫无用处，必须包含完整签名     |
| "用'处理相关逻辑'概括就行"              | 这种描述等于没说，必须具体描述处理什么输入、产出什么输出 |
| "这个模块是内部的，外部不会调用"        | internal 包也有明确的调用者，contract 描述的是模块间接口 |
| "先跳过这个复杂模块，生成其他的"        | 每一个被要求生成的模块都必须生成，不能跳过               |
```

**2. 在 `pack` 的 refine 模板中增加约束**

```markdown
## ⚠ 评分时不要犯这些错误

| 你可能会想                                 | 为什么这是错的                                 |
| ------------------------------------------ | ---------------------------------------------- |
| "这个文件和任务关键词没有直接匹配，给低分" | 间接依赖（被调用的工具函数）同样重要           |
| "测试文件和任务无关，给低分"               | 测试文件揭示了接口的使用方式和边界条件         |
| "配置文件不是代码，给低分"                 | 配置文件定义了运行时行为，经常是实现的关键约束 |
| "types.go 只是类型定义，低相关"            | 类型定义是理解数据流的关键，几乎总是高度相关的 |
```

### 改进三：生成后自审查步骤

**优先级：🟡 中（提升质量，需要额外 LLM 调用成本）**

#### 问题现状

当前 `init --scan` 的 9 阶段流水线中，每个阶段的 LLM 输出被直接采纳写入文件。没有独立的审查步骤来检查生成质量。导致的问题包括：

- 模块契约遗漏关键接口
- 架构层级划分不准确
- 依赖关系图存在遗漏

#### 借鉴思路

Superpowers 的两阶段审查模型：每次实现后，分别由**规格合规审查员**和**代码质量审查员**进行独立审查。特别关键的设计是审查员被告知"实现者的报告可能不完整，不要信任报告"。

#### 实现方案

**1. 新增审查阶段到 scan 流水线**

将 9 阶段扩展为 11 阶段（在关键生成步骤后插入审查）：

```
原流水线：
S1(文件扫描) → S2(摘要提取) → S3(关键文件选择) → S4(内容读取) →
S5(LLM分析) → S6(清单生成) → S7(架构/约定生成) → S8(模块契约生成) → S9(依赖图生成)

新流水线：
S1(文件扫描) → S2(摘要提取) → S3(关键文件选择) → S4(内容读取) →
S5(LLM分析) → S6(清单生成) → S7(架构/约定生成) → S7.5(架构自审查) ★新增 →
S8(模块契约生成) → S8.5(契约自审查) ★新增 → S9(依赖图生成)
```

**2. 新增审查提示词模板**

```

└── init_scan_review_contract.tmpl      ★新增

**3. 在 `internal/generator/engine.go` 中增加审查逻辑**

// 架构审查（S7.5）
func (e *Engine) reviewArchitecture(ctx context.Context, arch string, dirTree string) (string, error) {
    reviewPrompt := templates.Render("init_scan_review_architecture.tmpl", map[string]string{
        "DirectoryTree":          dirTree,
        "GeneratedArchitecture": arch,
    })

    resp, err := llm.ChatWithRetry(e.client, ChatRequest{
        Messages: []Message{{Role: "user", Content: reviewPrompt}},
    }, 2, nil)

    if strings.Contains(resp.Content, "NEEDS_REVISION") {
        // 提取修改建议，重新生成
        return e.regenerateArchitecture(ctx, arch, resp.Content)
    }
    return arch, nil
}
```

### 改进五：模块契约的触发条件字段

**优先级：🟡 中（提升 pack 精度）**

#### 问题现状

当前 `MODULE_CONTRACTS/*.yaml` 的结构是：

**2. 生成时让 LLM 填充触发条件**

````markdown

**3. 在 `packer/matcher.go` 中优先使用 `relevant_when` 进行匹配**

```go
func (m *Matcher) MatchContracts(task string, contracts []schema.ModuleContract) []schema.ModuleContract {
    var matched []schema.ModuleContract
    for _, c := range contracts {
        // 优先用 relevant_when 匹配
        if len(c.RelevantWhen) > 0 {
            for _, trigger := range c.RelevantWhen {
                if m.semanticMatch(task, trigger) {
                    matched = append(matched, c)
                    break
                }
            }
        } else {
            // 降级到关键词匹配（兼容旧版 contract）
            if m.keywordMatch(task, c.Description) {
                matched = append(matched, c)
            }
        }
    }
    return matched
}
````

### 改进六：模型分级策略

**优先级：🟢 低（优化成本，非功能性改进）**

#### 问题现状

当前所有 LLM 调用（scan 的 9 个阶段、pack 的 refine、update 的各阶段）都使用同一个配置的模型。但实际上不同阶段对模型能力的需求差异很大。

#### 借鉴思路

Superpowers 为子代理任务推荐不同级别的模型（机械任务用便宜模型，架构任务用强模型）。

#### 实现方案

**1. 在配置中增加模型分级**

| S9 依赖图生成   | standard | 需要理解模块关系         |
| Pack refine     | fast     | 相关度评分是机械性判断   |

## 第四部分：总结

Superpowers 和 Kontext 是 AI 辅助开发生态中**互补的两半**：

- **Superpowers** = 行为编译器：告诉 AI **怎么做事**（流程、纪律、质量门控）
- **Kontext** = 上下文编译器：告诉 AI **了解什么**（项目结构、约定、模块接口）

从 Superpowers 借鉴的核心经验不是它的代码（它几乎没有代码），而是它的**设计哲学**：

1. **预判 AI 的偷懒行为并显式封堵**（反合理化工程）
2. **不信任单次生成，用独立审查来保证质量**（两阶段审查）
3. **为 AI 消费优化元数据格式**（CSO / relevant_when）
4. **让集成零摩擦**（session hook 自动注入）
5. **分级使用资源**（便宜模型做机械活，强模型做判断）
6. **对提示词本身做质量保障**（prompt TDD / 回归测试）

这些经验的共同主题是：**把 AI 当成一个需要管理的初级工程师，而不是一个完美的自动化工具**。
