# Pack 流水线增强：LLM 驱动的任务引用文件识别与智能文件命名

## Context

当用户运行 `kontext pack "实现 docs/foo.md 的改进二"` 时，任务描述中引用了 `docs/foo.md`，但当前 pack 流水线无法识别和读取它。原因：`collector.go` 的 `collectSourceFiles()` 只收集代码文件（`.go`, `.py` 等），非代码文件被完全忽略。

本方案通过 **LLM 识别任务中的文件路径 + 文件存在性验证 + 指数退避重试** 解决此问题，同时让 LLM 为输出文件生成语义化文件名。

## 实现方案

### 1. 新增 `templates/pack_extract_files_system.tmpl` — 文件提取提示词

系统提示词，指导 LLM 从任务描述中提取文件路径并生成文件名：

```
你是一个文件路径提取专家。

## 任务

从用户的任务描述中提取所有被引用的文件路径，并为该任务生成一个简洁的中文概括文件名。

## 要求

1. 提取任务描述中显式或隐式提到的文件路径（如 `docs/foo.md`、`internal/packer/engine.go`）
2. 只提取看起来像文件路径的字符串（包含 `/` 或 `\` 分隔符，且有文件扩展名）
3. 不要猜测或编造不存在于任务描述中的路径
4. 为任务生成一个简洁的中文概括名称（不超过20个字，用于文件命名）

## ⚠ 不要犯这些错误

| 你可能会想 | 为什么这是错的 |
|---|---|
| "任务里没有明确路径，我猜一个" | 只提取任务文本中实际出现的路径，不要猜测 |
| "路径看起来不完整，我补全它" | 原样返回任务中出现的路径，由调用方验证 |
| "文件名概括写长一点更准确" | 文件名用于命名，必须简洁，不超过20字 |
```

### 2. 新增 `templates/pack_extract_files_user.tmpl` — 用户提示词模板

```
任务描述：
{{.Task}}

项目目录结构（用于辅助判断路径合理性）：
{{.DirectoryTree}}

请提取文件路径并生成文件名概括。
```

### 3. 在 `templates/templates.go` 中嵌入新模板

```go
//go:embed pack_extract_files_system.tmpl
var PackExtractFilesSystem string

//go:embed pack_extract_files_user.tmpl
var PackExtractFilesUser string
```

### 4. 新增 `internal/packer/extractor.go` — 核心提取逻辑

```go
// ExtractResult 是 LLM 文件提取的结构化输出
type ExtractResult struct {
    FilePaths   []string `json:"file_paths"`
    FilenameTip string   `json:"filename_tip"`
}

// ExtractReferencedFiles 调用 LLM 识别任务中引用的文件路径，
// 验证文件存在性，读取内容，返回 path->content 映射和文件名建议。
// 如果 LLM 返回的路径不存在，使用指数退避重试（最多 3 次），
// 每次重试时将不存在的路径反馈给 LLM 让其修正。
func ExtractReferencedFiles(client llm.Client, task string, projectDir string, dirTree string, onRetry func(int, error, time.Duration)) (map[string]string, string, error)
```

核心流程：
1. 渲染提示词模板，调用 `llm.ChatStructuredWithRetry` 获取 `ExtractResult`
2. 对每个返回的路径用 `os.Stat` 验证文件存在性
3. 如果有路径不存在，构造包含错误信息的新 prompt，指数退避后重试
4. 对存在的文件读取内容（上限 50KB），返回 `map[path]content` 和文件名建议
5. 如果 LLM 未返回任何文件路径，返回空 map（不报错，正常降级）

### 5. 修改 `internal/packer/collector.go` — `CandidateContext` 增加字段

```go
type CandidateContext struct {
    // ... 现有字段 ...
    ReferencedDocs map[string]string  // 任务描述中引用的文件 path->content
}
```

### 6. 修改 `internal/packer/engine.go` — 插入新阶段 + 文件名

流水线从 7 阶段扩展为 8 阶段，在阶段 2（粗筛）之后插入阶段 3（文件提取）：

```
原: S1(加载配置) → S2(粗筛) → S3(LLM精筛) → S4(读取) → S5(构建) → S6(LLM生成) → S7(保存)
新: S1(加载配置) → S2(粗筛) → S3(引用文件提取) → S4(LLM精筛) → S5(读取) → S6(构建) → S7(LLM生成) → S8(保存)
```

- S3 调用 `ExtractReferencedFiles`，结果存入 `ctx.ReferencedDocs`
- LLM 返回的 `filename_tip` 存入 `Engine.FilenameHint`（如果用户未指定 hint）

### 7. 修改 `internal/packer/prompt_template.go` — `TemplateData` 增加字段

```go
type TemplateData struct {
    // ... 现有字段 ...
    ReferencedDocs string  // 任务引用的文档内容
}
```

### 8. 修改 `internal/packer/builder.go` — 填充 `ReferencedDocs`

从 `ctx.ReferencedDocs` 渲染为 markdown（不用代码块包裹，因为内容可能是 markdown）。

### 9. 修改 `templates/llm_user_prompt.tmpl` — 新增模板段

在 `## Task` 之后插入：

```
{{if .ReferencedDocs}}## Task-Referenced Documents

以下文档在任务描述中被直接引用，请仔细阅读作为需求上下文：

{{.ReferencedDocs}}
{{end}}
```

### 10. 修改 `internal/promptdoc/writer.go` — 文件名格式调整

将 `GenerateFilename` 的时间戳格式改为 `YYYYMMDDHHMMSS`（当前是 `YYYYMMDD-HHMMSS`）：

```go
ts := time.Now().Format("20060102150405")
return fmt.Sprintf("%s-%s.md", ts, slugValue)
```
