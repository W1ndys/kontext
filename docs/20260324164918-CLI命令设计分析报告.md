# Kontext CLI 命令设计分析报告

> 分析时间：2026-03-24
> 分析范围：`cmd/` 目录下全部命令实现（root / init / pack / validate / update / config）

---

## 一、当前命令总览

| 命令 | 用途 | 代码行数 | 复杂度 |
|------|------|----------|--------|
| `kontext init` | 初始化 `.kontext/` 目录 | ~1980 行 | 极高 |
| `kontext pack` | 打包上下文为 Markdown Prompt | ~180 行 | 中 |
| `kontext validate` | 校验 YAML 配置 | ~35 行 | 低 |
| `kontext update` | 检测变更并更新物料 | ~400 行 | 中高 |
| `kontext config` | 管理全局 LLM 配置 | ~480 行 | 中 |
| `kontext -v` | 查看版本 | Cobra 内置 | 低 |

---

## 二、核心问题分析

### 问题 1：`init.go` 严重膨胀（最高优先级）

**现状：** `cmd/init.go` 单文件约 1980 行，是整个 `cmd/` 包中其他所有文件行数总和的约 2 倍。

**根因：** 以下职责全部堆砌在一个文件中：
- 3 种初始化模式的入口调度（交互式 / AI / scan）
- 9 阶段 scan 流水线的完整编排逻辑
- 缓存检查点恢复逻辑（`runScanInitFromCheckpoint`、`runScanInitResumeEarlyStages`）
- TUI 辅助函数（进度条、spinner、文件列表打印等）
- 默认模板常量（~120 行纯字符串）
- 辅助工具函数（`extractModulesFromArch`、`localAnalyzeFiles`、`isSourceFile` 等）

**影响：**
- 可读性差，新贡献者难以快速定位逻辑
- 缓存恢复路径与全新执行路径存在大量重复代码（阶段 1-5 的执行逻辑在 `runScanInitFresh` 和 `runScanInitResumeEarlyStages` 中几乎完全重复）
- TUI 工具函数与业务逻辑耦合，无法复用

**建议拆分方案：**

```
cmd/
  init.go                  # 入口调度 + flag 定义（~80 行）
  init_interactive.go      # 交互式 / AI 初始化（~100 行）
  init_scan.go             # scan 流水线编排（~500 行，合并消除重复）
  init_templates.go        # 默认模板常量（~120 行）
  init_helpers.go          # TUI 辅助函数（spinner / 进度条 / 文件列表）（~200 行）
```

同时，应将阶段 1-5 的执行逻辑统一为一个函数，通过 `startStage` 参数控制起始点，彻底消除 `runScanInitFresh` 与 `runScanInitResumeEarlyStages` 之间的重复。

---

### 问题 2：scan 流水线中断恢复的代码重复

**现状：** `runScanInitFresh`（248-542 行）和 `runScanInitResumeEarlyStages`（1214-1491 行）包含几乎相同的阶段 1-5 执行逻辑，只是后者多了缓存加载和条件跳过。

**影响：** 任何阶段逻辑的修改都需要在两处同步更新，容易遗漏。

**建议：** 统一为单一函数：

```go
func runScanPipeline(client llm.Client, cp *cache.Checkpoint, startStage int) error {
    // 阶段 1-5：根据 startStage 决定跳过还是执行
    // 阶段 6-9：复用现有 executeScanStages6to9
}
```

---

### 问题 3：`init` 与 `update` 缺乏子命令拆分

**现状：**
- `kontext init` 通过 `--scan` flag 切换完全不同的执行路径（交互式 vs 自动扫描），两条路径的复杂度和行为差异巨大
- `kontext update --dry-run` 的行为（只打印报告）与 `update` 的默认行为（执行更新）截然不同

**建议：**
- `init` 考虑拆分为子命令方案（可选，需权衡用户习惯）：
  ```
  kontext init          # 交互式（保持不变）
  kontext init scan     # 自动扫描（替代 --scan）
  kontext init scan --fresh / --resume
  ```
  这种方式让帮助信息更清晰，flag 不会交叉污染（如当前 `--fresh` / `--resume` 只在 `--scan` 下有效，但 `kontext init --fresh` 不会报错，只是被静默忽略）

- `update` 的 `--dry-run` 可考虑独立为 `kontext update status` 或 `kontext diff`（可选）

---

### 问题 4：flag 有效性缺乏校验

**现状：**
- `kontext init --fresh` 不带 `--scan` 时，`--fresh` 被静默忽略
- `kontext init --resume` 不带 `--scan` 时，`--resume` 被静默忽略

**影响：** 用户输入无效组合时没有任何反馈，可能导致困惑。

**建议：** 在 `RunE` 开头添加校验：

```go
if !scanFlag && (freshFlag || resumeFlag) {
    return fmt.Errorf("--fresh 和 --resume 仅在 --scan 模式下有效")
}
```

---

### 问题 5：`pack` 缺少 `--output` / `-o` 选项

**现状：** 输出路径完全由 `engine.Pack()` 内部决定，用户无法指定。

**影响：** 不利于 CI/CD 集成和脚本化使用。

**建议：** 增加 `--output` / `-o` 选项，允许用户指定输出文件路径或目录。

---

### 问题 6：`validate` 功能过于单薄

**现状：** `validate` 只调用 `schema.ValidateBundle()` 做 YAML 格式校验。

**可扩展方向（非紧急，按需实施）：**
- `--fix`：自动修复可修复的问题（如缺失字段补默认值）
- `--verbose`：显示每个文件的详细校验结果
- 校验模块契约中 `depends_on` 引用的模块是否确实存在

---

### 问题 7：全局状态和硬编码路径

**现状：**
- `.kontext` 路径在多个命令中硬编码为字符串字面量
- `projectDir := "."` 在 `init`、`pack`、`update` 中分别硬编码
- 全局变量（`scanFlag`、`freshFlag`、`packFromFile` 等）分散在各文件中

**影响：** 未来如需支持 `--dir` / `--project` 指定非当前目录的项目，改动范围较大。

**建议：**
- 将 `.kontext` 和 `.` 提取为 root 级常量或 persistent flag（`--kontext-dir`、`--project-dir`）
- 当前不必立即实现，但在设计新命令时应避免继续硬编码

---

### 问题 8：`config` 的交互式向导与 TUI 组件偏重

**现状：** `config.go` 包含完整的 Bubble Tea 模型选择器（~120 行），交互式向导使用 `bufio.Reader` 读取输入。

**观察：**
- 两种交互方式（Bubble Tea TUI + bufio 原始输入）混用在同一个流程中，体验不一致
- 模型选择器代码占 `config.go` 约 25%，但只在一处使用

**建议：**
- 将 Bubble Tea 模型选择器提取到 `cmd/config_tui.go` 或 `internal/tui/` 包中
- 长期考虑将 `bufio.Reader` 交互也统一为 Bubble Tea 组件

---

## 三、设计亮点（值得保持）

1. **`pack` 命令的多输入源设计** — 支持位置参数、`--from-file`、stdin（`-`）和交互式提示，覆盖全场景
2. **`update` 的 detect → plan → confirm → execute 四阶段流程** — 安全且透明
3. **`config` 的子命令结构** — `set` / `get` / `list` 清晰规范，默认无参数启动向导也很友好
4. **结构化日志贯穿全局** — 每个命令路径都有独立的 logger，日志字段语义明确
5. **中英双语帮助信息** — 国际化意识好，降低非中文用户使用门槛

---

## 四、优先级排序建议

| 优先级 | 改进项 | 工作量 | 收益 |
|--------|--------|--------|------|
| P0 | `init.go` 拆分 + 消除重复代码 | 中 | 可维护性大幅提升 |
| P1 | flag 有效性校验（`--fresh`/`--resume` 无 `--scan`） | 极小 | 防止用户困惑 |
| P1 | `pack` 增加 `--output` 选项 | 小 | CI/CD 友好 |
| P2 | 路径常量提取（`.kontext`、`.`） | 小 | 为未来扩展打基础 |
| P2 | `config` TUI 组件独立拆分 | 小 | 代码整洁 |
| P3 | `init` 子命令化（`init scan`） | 中 | 命令语义更清晰，但涉及用户习惯变更 |
| P3 | `validate` 功能增强 | 中 | 锦上添花 |

---

## 五、总结

Kontext CLI 的命令结构整体合理，`config` 的子命令设计、`pack` 的多输入源、`update` 的安全流程都是优秀实践。最突出的问题是 **`init.go` 的严重膨胀和代码重复**，建议作为首要改进项。其次是 **flag 校验缺失** 和 **`pack` 缺少输出路径控制** 这两个低成本高收益的改进。
