# 统一 LLM 输出为 JSON 格式方案

## 背景

当前项目 LLM 调用存在两种输出路径：

1. **`ChatStructured`**：通过 OpenAI `response_format` + JSON Schema 约束输出，`json.Unmarshal` 反序列化 → **可靠**
2. **`ChatYAML`**：无 API 级别约束，纯粹靠提示词引导 LLM 输出 YAML，`yaml.Unmarshal` 反序列化 → **不可靠，需要回退和重试**

此外，`update` 命令使用 `Chat`（纯文本）让 LLM 直接返回 YAML 文本，校验后写入文件。

**问题：** LLM 天生适合返回 JSON（API 原生支持 JSON Schema 强制约束），要求返回 YAML 完全依赖提示词工程，容易出格式错误，导致复杂的回退/重试逻辑。

**目标：** 所有 LLM 调用统一使用 JSON 作为中间格式，需要 YAML 时在应用层转换。

## 改造方案

### 第一阶段：LLM 接口层改造

#### 1.1 废弃 `ChatYAML` 方法

**涉及文件：**
- `internal/llm/types.go` — 从 `Client` 接口移除 `ChatYAML`
- `internal/llm/openai.go` — 删除 `ChatYAML` 实现和 `stripYAMLCodeBlock` 辅助函数
- `internal/llm/retry.go` — 删除 `ChatYAMLWithRetry`
- `internal/llm/errors.go` — 移除 YAML 相关的错误检测逻辑（如有）

**理由：** `ChatYAML` 无 API 级约束，依赖提示词工程，不可靠。统一用 `ChatStructured`（JSON Schema 约束）替代。

#### 1.2 `Client` 接口变更

```go
// 改造前
type Client interface {
    Generate(req *GenerateRequest) (*GenerateResponse, error)
    Chat(req *ChatRequest) (*ChatResponse, error)
    ChatStream(req *ChatRequest, onChunk func(string) error) (*ChatResponse, error)
    ChatStructured(req *ChatRequest, schemaName string, out any) (*ChatResponse, error)
    ChatYAML(req *ChatRequest, out any) (*ChatResponse, error)  // 删除
    ListModels() ([]string, error)
}

// 改造后
type Client interface {
    Generate(req *GenerateRequest) (*GenerateResponse, error)
    Chat(req *ChatRequest) (*ChatResponse, error)
    ChatStream(req *ChatRequest, onChunk func(string) error) (*ChatResponse, error)
    ChatStructured(req *ChatRequest, schemaName string, out any) (*ChatResponse, error)
    ListModels() ([]string, error)
}
```

### 第二阶段：数据类型调整

#### 2.1 核心类型不变

`internal/generator/types.go` 中的类型已同时有 `yaml` 和 `json` tag，无需修改结构定义。

#### 2.2 关键语义变化

对于 `GeneratedYAML`、`SingleFileYAML`、`ModuleContractYAML` 这三个类型，其字符串字段（如 `Content`、`ProjectManifest` 等）的值将从"YAML 文本"变为"YAML 文本"不变——但获取方式改为：LLM 返回 JSON 结构 → `json.Unmarshal` 反序列化 → 字段值仍是 YAML 字符串。

**关键设计决策：** LLM 应返回 JSON 对象，其中需要写入 .yaml 文件的字段值仍然是 YAML 字符串（JSON 的 string 字段内嵌 YAML 文本）。这样下游写入文件的逻辑完全不受影响。

### 第三阶段：调用站点改造

#### 3.1 `GenerateStructuredYAML`（engine.go:240-259）

**当前：** `client.ChatYAML(req, &structured)` → 回退到 `generateYAMLWithRetry`
**改为：** `client.ChatStructured(req, "generated_yaml", &structured)` → 回退到 `Chat` + `json.Unmarshal`

```go
// 改造后
func GenerateStructuredYAML(client llm.Client, systemPrompt, userMsg string) (*GeneratedYAML, error) {
    req := &llm.ChatRequest{
        Messages: []llm.Message{
            {Role: "system", Content: systemPrompt},
            {Role: "user", Content: userMsg},
        },
    }

    var structured GeneratedYAML
    if _, err := client.ChatStructured(req, "generated_yaml", &structured); err == nil {
        return &structured, nil
    } else {
        // 回退：纯文本 Chat + JSON 解析
        generated, fallbackErr := generateJSONWithRetry(client, systemPrompt, userMsg)
        if fallbackErr != nil {
            return nil, fmt.Errorf("JSON 结构化输出失败: %v；回退解析也失败: %w", err, fallbackErr)
        }
        return generated, nil
    }
}
```

#### 3.2 `GenerateSingleYAML`（engine.go:420-477）

**当前：** `client.ChatYAML(req, &structured)` → 回退到 `Chat` + `ParseSingleFileYAML`(YAML)
**改为：** `client.ChatStructured(req, "single_file_yaml", &structured)` → 回退到 `Chat` + `json.Unmarshal`

#### 3.3 `GenerateModuleContract`（engine.go:503-560）

**当前：** `client.ChatYAML(req, &structured)` → 回退到 `Chat` + `ParseModuleContractYAML`(YAML)
**改为：** `client.ChatStructured(req, "module_contract_yaml", &structured)` → 回退到 `Chat` + `json.Unmarshal`

#### 3.4 `GenerateModuleContractStream`（engine.go:633-724）

**当前：** `client.ChatStream` → `extractYAMLFromResponse`
**改为：** 流式输出场景不适合使用 JSON Schema 约束。有两个选择：
- **方案 A（推荐）：** 改用 `ChatStructured` 非流式调用，放弃流式输出。生成单个模块契约的耗时通常不长，流式 UX 改进有限。
- **方案 B：** 保留流式，但改提示词为 JSON 格式，完成后用 `json.Unmarshal` 解析。

**建议采用方案 A**，简化代码。如果用户体验要求必须流式，则采用方案 B。

#### 3.5 updater `generateYAMLContent`（executor.go:256-264）

**当前：** `ChatWithRetry` → 纯文本 YAML → `extractYAMLContent` → 直接写文件
**改为：** 使用 `ChatStructured` 返回 JSON 结构，再从中提取 YAML 内容。

具体步骤：
1. 定义 updater 专用的响应结构：

```go
// UpdatedYAML 是 LLM 更新制品时返回的 JSON 结构。
type UpdatedYAML struct {
    Content string `json:"content"` // 更新后的 YAML 文本
}
```

2. 将 `generateYAMLContent` 改为 `ChatStructured` 调用：

```go
func (e *Executor) generateYAMLContent(req *llm.ChatRequest, ...) (*llm.ChatResponse, string, error) {
    var result UpdatedYAML
    resp, err := llm.ChatStructuredWithRetry(e.client, req, "updated_yaml", &result, 3, nil)
    if err != nil {
        return nil, "", fmt.Errorf("调用 LLM 生成内容失败: %w", err)
    }
    return resp, result.Content, nil
}
```

3. 更新 `generateContent` 中的重试逻辑，将 YAML 格式错误提示改为 JSON 格式。

#### 3.6 回退解析函数改造（parser.go）

**当前：** 所有 `Parse*` 函数使用 `yaml.Unmarshal`
**改为：** 改用 `json.Unmarshal`，因为回退路径的 `Chat` 调用也将通过提示词引导返回 JSON

```go
// 改造前
func ParseGeneratedYAML(raw string) (*GeneratedYAML, error) {
    cleaned := stripCodeBlock(raw)
    var result GeneratedYAML
    if err := yaml.Unmarshal([]byte(cleaned), &result); err != nil {
        return nil, err
    }
    return &result, nil
}

// 改造后
func ParseGeneratedJSON(raw string) (*GeneratedYAML, error) {
    cleaned := stripCodeBlock(raw)
    var result GeneratedYAML
    if err := json.Unmarshal([]byte(cleaned), &result); err != nil {
        return nil, err
    }
    return &result, nil
}
```

同理改造：`ParseInterviewResponse`、`ParseAnalyzedFiles`、`ParseSelectedFiles`、`ParseSingleFileYAML`、`ParseModuleContractYAML`、`ParseModuleDependencyGraph`。

`stripCodeBlock` 中也应优先匹配 `json` 代码块。

## 风险与注意事项

1. **JSON 字符串中嵌入 YAML 文本的转义问题：** JSON 字符串需要转义换行符（`\n`）、引号等。LLM 通过 JSON Schema 约束时通常能正确处理，但需要在集成测试中验证多行 YAML 内容的正确性。

2. **JSON Schema 的 strict 模式限制：** OpenAI 的 `strict: true` 要求所有字段都必须在 schema 中定义。`GeneratedYAML` 中的 `module_contracts` 是 `map[string]string`，在 strict JSON Schema 中需要使用 `additionalProperties` 来描述。需验证 `invopop/jsonschema` 对 `map[string]string` 的 schema 生成是否兼容 OpenAI strict 模式。

3. **流式输出改造：** `GenerateModuleContractStream` 改为非流式后，用户在生成多个模块契约时将看不到实时进度文字。但并行生成 + 进度回调仍能提供良好的 UX。

4. **向后兼容：** 此改造涉及 LLM 接口变更（删除 `ChatYAML`），如果有外部使用者依赖此接口，需要同步更新。当前项目为 CLI 工具，接口仅内部使用，风险可控。

5. **测试验证：** 改造后需运行 `go test ./...` 确保编译通过，并对 `init`、`pack`、`update` 三个命令进行端到端验证。
