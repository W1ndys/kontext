# 制品存储格式从 YAML 迁移到 JSON

## 背景

Kontext 的结构化制品（PROJECT_MANIFEST、ARCHITECTURE_MAP、CONVENTIONS、模块契约）当前以 YAML 格式存储在 `.kontext/` 目录下。LLM 生成的 YAML 内容经常出现格式错乱（缩进不一致、多余空格、特殊字符未转义），导致解析失败。虽然已实现 `NormalizeYAML`（unmarshal → marshal 标准化），但 YAML 的 unmarshal 阶段本身就可能因 LLM 输出的格式问题而失败，尤其是包含代码示例的 CONVENTIONS 制品。

**决策：将所有制品的存储格式从 YAML 迁移到 JSON。**

### 迁移优势

- JSON 格式严格、无歧义，不存在缩进敏感问题
- LLM 原生支持 JSON Schema 结构化输出，生成质量远高于 YAML
- `json.Marshal` 输出格式完全一致，无需额外标准化
- 消除 `yaml.Unmarshal` 解析失败的根因

### 与已有方案的关系

`docs/20260327160000-统一LLM输出为JSON格式方案.md` 关注的是 **LLM 调用层**（ChatYAML → ChatStructured），本文档关注的是 **制品存储层**（.yaml 文件 → .json 文件），两者互补。

#### writeJSONFile 实现

```go
func writeJSONFile(path string, content interface{}) error {
    data, err := json.MarshalIndent(content, "", "  ")
    if err != nil {
        return fmt.Errorf("序列化 %s 失败: %w", filepath.Base(path), err)
    }
    if err := fileutil.WriteFile(path, data); err != nil {
        return fmt.Errorf("写入 %s 失败: %w", path, err)
    }
    return nil
}
```

#### GenerateSingleJSON 核心逻辑变化

当前流程：LLM → JSON envelope `{"content": "YAML text"}` → 提取 content → yaml.Unmarshal 验证 → 写入 .yaml

新流程：LLM → JSON Schema 结构化输出（直接返回制品结构体）→ json.Marshal → 写入 .json

**关键设计变化**：不再让 LLM 返回 `{"content": "一段文本"}` 的间接格式，而是直接让 LLM 返回制品的完整 JSON 结构。例如生成 PROJECT_MANIFEST 时，LLM 直接返回：

```json
{
  "project": {
    "name": "Kontext",
    "one_line": "AI 原生的上下文编译器",
    ...
  },
  "tech_stack": { ... },
  "scale": { ... },
  "status": { ... }
}
```

这样可以直接 `json.Unmarshal` 为 `schema.ProjectManifest` 结构体，再 `json.MarshalIndent` 写入文件。

#### extractModulesFromArchitecture 改造

```go
func extractModulesFromArchitecture(archJSON string) []string {
    var arch schema.ArchitectureMap
    if err := json.Unmarshal([]byte(archJSON), &arch); err != nil {
        return nil
    }
    // ... 其余逻辑不变
}
```

// ...

    return fmt.Errorf("格式化 JSON 失败: %w", err)
}
return fileutil.WriteFile(targetPath, formatted)

## 风险与注意事项

1. **向后兼容**：用户已有的 `.kontext/*.yaml` 文件将不再被识别。建议：
   - 在 loader 中添加临时兼容逻辑：优先读 `.json`，回退读 `.yaml`
   - 或提供 `kontext migrate` 命令一键转换
   - 在 README changelog 中明确标注为 breaking change

2. **config.yaml 不受影响**：全局 LLM 配置 `~/.kontext/config.yaml` 保持 YAML 格式不变。

3. **JSON 可读性**：JSON 不支持注释，但制品文件主要由程序生成和消费，注释需求不高。使用 `MarshalIndent` 保证格式美观。

4. **LLM 模板数据字段名**：部分模板使用 `{{.CurrentYAML}}`、`{{.Part1YAML}}` 等字段名，需要同步修改调用方传入的 template data 结构体或 map 的 key。

5. **测试覆盖**：`internal/schema/` 有现有测试，迁移后需更新测试数据从 YAML 改为 JSON。
